# GIS 引擎研发汇报：从需求到平台底座的演进

## 1. 需求破冰与早期选型（2月27日 - 3月17日）

### 1.1 从零理解 GIS 与初次选型（2月27日）
首次接到 GIS 相关需求时，团队对地理信息系统（GIS）及产品形态缺乏具象概念。我们从“GIS”和“MapBox”这两个名词切入：
*   **需求理解**：通过调研市场主流 GIS 系统，初步描绘出产品需要具备的地图渲染和空间数据管理能力。
*   **技术选型痛点**：原计划采用知名的 MapBox，但调研发现其已更改开源协议，且对中国大陆地区支持受限。
*   **解决方案**：转向纯开源方案。在对比 OpenLayers、Leaflet 和 MapLibre GL 后，最终敲定 **MapLibre GL**。
    *   **选型理由**：高性能（WebGL 渲染）、学习曲线适中、完全开源免费、且易与现有前端技术栈融合。
*   **🤖 AI 赋能**：在面对完全陌生的领域时，AI 发挥了巨大的扫盲作用。它快速帮我们总结了各框架的能力边界和生态状况，大幅缩短了技术调研时间。随后我们基于 MapLibre GL 快速输出了一个最小验证样例。

### 1.2 Demo 验证与需求对齐（3月3日 - 3月17日）
*   **3月3日**：完成第一版 Demo。不仅跑通了基础渲染，还尝试引入 `@indoorequal/vue-maplibre-gl` 测试 Vue 组件化方案，探索更现代的开发体验。
*   **3月17日**：经过多轮需求对齐会议，明确了自研 GIS 引擎的核心目标和研发方向，产品雏形（需要什么图层、什么绘制能力）基本清晰。

## 2. 核心架构敲定与开发启动（3月20日）

在明确需求后，原有的简单组件库已经无法满足复杂的业务交互，我们进行了更深度的技术论证，敲定了最终的核心技术栈：
*   **底层渲染**：`MapLibre GL`（提供高性能矢量瓦片与基础地图能力）
*   **绘制引擎**：`TerraDraw`（专攻点、线、面等复杂图形的业务绘制与交互）
*   **空间计算**：`Turf.js`（提供距离、面积、质心等精准几何计算）

### 🚧 遇到问题与解决思路
*   **痛点**：如何快速验证这些独立框架能否完美融合？
*   **解决**：我们没有直接盲写代码，而是利用 AI 直接阅读这些开源库的 GitHub 源码和官方 Demo。AI 极大地提高了阅读源码的效率，帮我们快速验证了“MapLibre + TerraDraw + Turf.js”组合架构的可行性。
*   **关键动作**：在正式编码前，我们**优先利用 AI 梳理并建立了项目专属的本地知识库**。这不仅让开发人员快速上手，更重要的是让 AI 助手在后续开发中能直接复用这些上下文，避免了反复联网搜索，代码生成的准确率得到质的飞跃。

## 3. 项目演进与核心里程碑（3月23日 - 4月13日）

整个开发过程分为两个明确的阶段：**先在业务项目 (`tkekmpms-imms-f`) 中打磨功能，再抽象下沉为公共底座 (`vue-maplibre-kit`)**。

### 📍 3月23日-26日：基础能力落地与初步解耦
*   **动作**：完成了地图组件的初步集成，落地了悬浮高亮、点击弹窗、要素闪烁等基础交互，并跑通了早期的绘图与测量能力。
*   **抽象动作**：随着功能增加，核心组件迅速臃肿。我们果断将绘图/测量控件的样式配置、要素属性更新逻辑抽离为独立函数。这为后续的“统一属性管理”打下了第一根桩。

### 📍 3月27日-30日：第一次大型重构 —— 终结分裂，统一交互体系
*   **遇到的问题**：业务开发极度痛苦。普通地图要素和 TerraDraw 绘制要素的底层机制不同，导致它们的交互（如 hover、选中）是两套完全平行的系统，业务代码里塞满了恶心的 `if-else` 适配逻辑。
*   **重构理念**：必须终结这种分裂感，把所有的交互行为收口到一个统一的体系下。
*   **实际效果**：业务页面彻底告别了碎片化的事件分支。通过顶层事件托管与局部覆写结合，无论是普通图层还是绘制要素，拿数据、配样式的体验终于完全一致。

### 📍 3月31日-4月2日：视觉表现升级与临时预览托管
*   **动作**：补齐了强大的线装饰功能（连续纹理、拉伸覆盖等）。
*   **抽象动作**：将业务中高频出现的“临时巷道预览”需求，从零散的页面代码中提取出来，升级成了标准化的底层托管机制。业务只需传入数据和样式，底层自动编排图层。

### 📍 4月7日-9日：第二次大型重构 —— 成立公共库，引入插件架构
*   **重构契机与理念**：由于下班在家无法直连公司内网环境，为了不阻塞进度，我们新开了一个独立的工程库。这也倒逼我思考：如何保证这个抽离出来的底座足够轻量？答案是——不能把高级功能全塞进核心，必须设计一套标准规范的插件系统。
*   **实际效果**：正式抽离出 `vue-maplibre-kit`，项目完成了从“业务功能堆砌”到“平台级公共库”的质变。核心容器保持极简，而多选、高级交互、草稿预览等复杂能力，全部变成了即插即用的独立插件。

### 📍 4月10日-12日：第三次大型重构 —— 铺设门面层（Facade），统一数据契约
*   **重构契机与理念**：当尝试将公共库回迁至具体业务项目时，发现调用路径和 API 适配非常繁琐。这暴露出一个核心痛点：业务端依然在直面 MapLibre 原生的底层接口（source、layer 等），心智负担极重；且修改要素属性时缺乏规范，数据容易脏乱。
*   **解决方案**：我们大面积铺设了“门面层（Facade）”（如业务源门面、要素查询/动作门面、统一属性编辑门面）。
*   **实际效果**：业务层开发变得极其简单！开发者只需调用类似“查什么、改什么、高亮什么”的语义化 API，无需触碰底层原生方法。普通要素和绘制要素的写回规矩被完全统一，力求让项目真正成长为一个数据安全、高度可控的 GIS 业务底座。

## 4. AI Coding 在项目中的核心价值总结

整个 GIS 引擎从零到一的建设中，AI Coding 工具深度参与并发挥了不可替代的作用。本项目 `src/` 目录下核心源码高达 **2.4万行**，期间经历的数次大型重构删改代码超过 **1万行**，整个开发过程累计消耗了约 **5亿 Token**。如果没有 AI 的协助，这种规模的架构演进和代码重写，单人完成至少需要半年时间。

AI 的核心贡献体现在：
1.  **破冰与选型加速**：在业务初期，快速总结陌生领域的技术生态，大幅缩短技术调研和框架可行性验证的时间。
2.  **知识库驱动的精准生成**：通过建立本地项目知识库让 AI 学习，AI 能够深刻理解项目独特的插件架构和门面设计，生成的代码准确率极高，免去了繁琐的上下文提示。
3.  **重构与代码审查**：在三次大型重构中，AI 承担了大量繁重的重构代码迁移和逻辑适配工作，并在开发过程中持续进行代码漏洞审查和单元测试编写，极大地保障了重构过程中的系统稳定性。
