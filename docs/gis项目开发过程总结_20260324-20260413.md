## 3. 里程碑总结

### 3.1 业务项目内先完成基础能力拆分与配置治理

代表提交：

- `855cfd2` `2026-03-25` 提取绘图与测量控件样式配置到独立文件

- `706301f` `2026-03-26` 将地图要素属性更新逻辑重构为独立函数

这一阶段的重点不是“多加功能”，而是先把地图大组件从不断堆逻辑的状态里拉出来，形成可维护的配置结构。绘图和测量控件的样式从页面和组件内部抽离出来后，后续业务调整样式、切换模式、扩展图层表现时，不再需要频繁改动核心地图初始化逻辑。

这次重构的作用主要有三点：

- 把控件样式从“散落在业务页面中的硬编码”变成“集中配置”。
- 为后续统一绘图交互、测量行为和业务覆写能力打基础。
- 让属性编辑能力具备独立演进空间，不再和地图渲染代码强耦合。

### 3.2 交互体系开始统一，普通图层与 TerraDraw 不再各管一套

代表提交：

- `e8eb156` `2026-03-27` 新增 TerraDraw 统一业务交互封装
- `9a8791e` `2026-03-27` 添加普通图层交互管理功能
- `ea0c871` `2026-03-27` 优化属性保存逻辑，支持地图要素与 TerraDraw 要素分流处理
- `2647e4d` `2026-03-29` 优化 TerraDraw 交互缓存与快照管理
- `1f353a2` `2026-03-30` 重构普通图层交互管理，支持顶层统一事件与局部覆盖
- `d529a3e` `2026-03-30` 为普通图层交互添加选中状态支持

这是第一轮真正意义上的“大重构”。此前地图普通图层、绘图控件、测量控件、业务页面事件之间的职责边界比较分散，而这一轮开始把它们纳入统一的交互设计。

这次重构的目的很明确：

- 让普通图层和 TerraDraw 的交互不再是两套平行体系。
- 让业务层能在“统一入口”下处理 hover、click、选中态、属性编辑等动作。
- 通过快照和缓存机制，稳定绘图态与业务态之间的数据读取。

这次重构带来的直接作用也非常明显：

- 地图页面不必再自己拼接大量交互事件分支。
- 顶层统一事件和局部覆盖并存，既能全局托管，也保留局部定制能力。
- 后续多选、门面层、属性治理能继续往上叠加，而不用推翻之前的设计。

代表提交：

- `9571fcb` `2026-03-30` 提取巷道区域与延长工具

- `7994757` `2026-03-31` 支持多数据源下托管预览线的正式来源追踪

这一阶段可以看作从“做出功能”转向“沉淀模式”的分水岭。临时巷道预览本来是非常具体的业务功能，但提交记录显示，开发没有停留在页面功能实现，而是持续把它抽成托管预览、扩展机制、来源追踪、样式覆写这些更稳定的公共能力。

这轮重构的目的：

- 把巷道预览从页面逻辑里剥离出来，减少业务代码直接操心临时 source、layer、状态切换。
- 让预览线、线廊、来源追踪等行为变成托管能力，而不是一次性业务代码。
- 为后续迁移到公共库中的插件机制提前验证边界。

这轮重构的作用：

- 业务页面只关心“什么时候预览、预览哪条线、用什么样式”，不再关心底层图层编排。
- 几何算法开始从具体业务语义中抽离，为后续中立工具化做准备。
- 后来 `vue-maplibre-kit` 中的 `lineDraftPreview` 插件，本质上就是这一阶段能力的库化延续。

## 4. 从业务工程到公共库：平台化重构主线

### 4.1 公共库正式建立，开发重心从“做页面”切换为“做底座”

代表提交：

- `eebd029` `2026-04-07` 初始化 Vue 3 + MapLibre GL 项目基础结构
- `cd1245c` `2026-04-07` 重构为公共库结构并引入插件系统
- `7c23964` `2026-04-07` 重构地图插件架构，将扩展迁移为独立插件
- `dac52d3` `2026-04-07` 导出 `AnyMapPluginDescriptor` 类型

这是整个时间段里最重要的一次架构跃迁。此前能力主要存在于业务仓库中，而从 `2026-04-07` 开始，项目明显改为以 `vue-maplibre-kit` 为中心进行公共能力沉淀。

这次大重构的核心目标：

- 把原先偏业务化的 GIS 实现重组为可发布、可复用的公共库结构。
- 通过 `core / composables / plugins / facades / shared / terradraw` 分层，明确能力边界。
- 把“扩展”升级为“插件”，让高级能力按需接入，而不是继续塞进地图核心组件。

这次重构的实际作用：

- `managedTunnelPreview` 这类业务扩展，开始被重新组织为更通用的插件能力。
- `geometry` 子路径把线延长、线廊生成等几何能力抽成中立工具，降低业务语义污染。
- 公共 API 不再围绕某一个页面组织，而是围绕“稳定出口”和“按需导入”设计。

如果说 `04-07` 完成的是“库的出生”，那么 `04-08` 到 `04-09` 完成的就是“库的稳定化”。开发重点从功能抽离转向宿主机制、类型安全、生命周期、服务边界这些平台底座问题。

这轮重构的目的：

- 让插件真正成为长期可扩展的正式机制，而不是另一种写法的业务扩展。
- 把多选、查询、服务状态等复杂能力收敛到插件和门面，不污染核心宿主。
- 解决 TerraDraw 初始化时序、监听依赖和单例校验等稳定性问题。

它的作用在于：

- 地图核心更加轻量，复杂功能以插件形式外接。
- 高级能力开始具备“可组合、可替换、可关闭”的平台特征。
- 后续业务层接入多选、吸附、草稿预览时，接入方式更统一。

### 4.3 门面层补齐，业务层开始真正“少碰底层”

代表提交：

- `30f567b` `2026-04-10` 新增地图业务源门面与要素查询/动作门面
- `b02dda4` `2026-04-10` 新增轻量业务图层组件与线草稿预览门面
- `d757c78` `2026-04-10` 重构要素闪烁与样式表达式 API
- `ef550d1` `2026-04-10` 添加切换选中要素示例样式的功能

这是第二轮非常关键的大重构，重点不是底层图形能力本身，而是业务调用方式。可以明显看出项目开始有意识地把 MapLibre 原生 source/layer 操作、要素查询、要素动作、草稿预览等能力包装为门面。

这轮重构的目的：

- 降低业务页面对 MapLibre/TerraDraw 细节的直接依赖。
- 把“查什么”“改什么”“高亮什么”“预览什么”变成更贴近业务语义的接口。
- 让常见业务图层接入尽量通过组件和 hooks 完成，而不是手写一堆底层调用。

这轮重构的作用：

- 业务层开发心智负担明显下降。
- 公共库的对外形态从“地图组件集合”进一步升级为“GIS 门面能力层”。
- 后续属性治理与统一编辑门面得以自然落位，而不是硬塞到业务示例里。

### 4.4 属性治理与统一属性编辑落地，数据边界开始成型

代表提交：

- `fcfa2a9` `2026-04-11` 新增属性治理与面板态解析能力
- `979e398` `2026-04-11` 明确属性写回模式语义并收紧删键契约
- `777e197` `2026-04-12` 移除废弃的 `FeaturePropertySaveMode` 及相关逻辑
- `305ee4b` `2026-04-12` 新增统一属性编辑门面并优化业务源性能

这是最近一次最成体系的重构，重点已经从“地图怎么画”转向“地图数据怎么治理”。这说明项目开始进入平台成熟期，关注点不再只是交互体验，而是属性写回、编辑契约、面板状态解析、普通图层与绘制要素的一致治理。

这轮重构的目的：

- 统一普通地图图层要素与 TerraDraw 绘制要素的属性编辑入口。
- 用更清晰的写回语义替代旧的保存模式，减少歧义和历史包袱。
- 把属性面板、数据更新、业务源管理串成一套可维护的数据治理链路。

这轮重构的作用：

- 后续业务做属性面板时，不必分别处理“普通要素”和“绘图要素”两套保存逻辑。
- 公共库对外的行为约束更加清晰，减少误用空间。
- 项目从“功能库”进一步靠近“可控、可维护的业务底座”。

## 5. 这段时间沉淀出的核心成果

结合两个仓库的演进，可以把这段时间的产出概括为以下几项：

## 6. 最终结论

这段时间的开发不是简单地“持续加功能”，而是经历了一条很清晰的技术演进路径：

1. 先在 `tkekmpms-imms-f` 中围绕真实业务场景快速验证能力。
2. 再把稳定能力抽离到 `vue-maplibre-kit` 中，形成公共库结构。
3. 随后继续通过插件化、门面化、属性治理和类型约束，把这套能力从“能用”提升到“可复用、可维护、可扩展”。

如果用一句话评价这一阶段的开发成果，可以概括为：

**项目已经从“业务页面里的 GIS 功能集合”，演进为“面向业务平台建设的 WebGIS 前端底座”。**
