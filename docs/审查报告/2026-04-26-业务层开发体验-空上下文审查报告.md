# 2026-04-26 业务层开发体验空上下文审查报告

## 审查背景

本次审查模拟“真实业务项目安装并引入 `vue-maplibre-kit` 后”的业务层开发者视角，重点评估业务层 API、门面入口和 `NGGI01` 到 `NGGI10` 示例是否足够简单、易上手，以及导入负担是否过重。

审查方式为只读核对，未修改源码、示例或文档之外的任何文件。结论基于实际读取到的入口文件、示例代码、插件出口和 `docs/problem-record.md` 内容。

## 总体结论

当前项目已经基本满足“业务层可以只依赖公开出口完成接入”的目标。`package.json` 已暴露主入口、`business`、`config`、`geometry` 和多个插件子入口；`NGGI01` 到 `NGGI10` 示例没有发现直接引用 `src/MapLibre/**`、`@/MapLibre/**` 或源码内部路径的情况。

主要问题不是能力缺失，而是业务层推荐路径还不够唯一：插件注册在 `vue-maplibre-kit/plugins/*`，插件状态读取既可以走 `useLineDraftPreview`、`useIntersectionPreview`、`useMapFeatureMultiSelect`，也可以走 `useBusinessMap().plugins.*`，同时 `useBusinessMap` 还保留了 `draft`、`intersection` 两个兼容别名。对新开发者来说，容易产生“应该记哪个入口”的疑问。

基础能力示例覆盖较完整：地图初始化、source/layer、运行时 source/layer、查找要素、属性编辑、弹窗、snap、line-draft、intersection、multi-select 均有对应示例。若后续希望进一步降低 npm 消费者心智负担，建议优先优化插件聚合门面和示例中的推荐写法。

## 建议优化

### 1. 统一插件注册与插件状态读取路径

涉及文件：

- `examples/views/NG/GI/NGGI08.vue`
- `examples/views/NG/GI/NGGI09.vue`
- `examples/views/NG/GI/NGGI10.vue`
- `src/MapLibre/facades/useBusinessMap.ts`

理由：

`NGGI08.vue` 同时从 `vue-maplibre-kit/business` 引入 `useBusinessMap`、`useLineDraftPreview`，又从 `vue-maplibre-kit/plugins/line-draft-preview` 引入 `createLineDraftPreviewPlugin`。`NGGI09.vue` 使用 `useIntersectionPreview` 加 `createIntersectionPreviewPlugin`。`NGGI10.vue` 使用 `useMapFeatureMultiSelect` 加 `createMapFeatureMultiSelectPlugin`。

这些写法都是公开 API，但从业务开发者视角看，需要同时理解“注册插件”和“读取插件状态”属于不同入口。建议专项示例尽量统一成 `createBusinessPlugins + useBusinessMap().plugins`，除非示例目标就是展示底层插件工厂。

### 2. 减少 `businessMap.plugins.lineDraft` 与 `businessMap.draft` 的混用

涉及文件：

- `src/MapLibre/facades/useBusinessMap.ts`
- `examples/views/NG/GI/NGGI06.vue`

理由：

`useBusinessMap` 返回值中同时存在：

- `businessMap.plugins.lineDraft`
- `businessMap.draft`
- `businessMap.plugins.intersection`
- `businessMap.intersection`

源码注释说明 `draft` 和 `intersection` 是兼容别名，但 `NGGI06.vue` 中同时展示了 `businessMap.intersection`、`businessMap.plugins.multiSelect`、`businessMap.draft`、`businessMap.plugins.lineDraft`。这会削弱“推荐入口”的明确性。

建议示例统一使用 `businessMap.plugins.*`，或者反过来只在业务文档中推荐短路径，把 `plugins` 定位为高级入口。

### 3. 新增插件聚合入口，降低 import 记忆成本

涉及文件：

- `package.json`
- `src/plugins/map-feature-snap.ts`
- `src/plugins/line-draft-preview.ts`
- `src/plugins/intersection-preview.ts`
- `src/plugins/map-feature-multi-select.ts`
- `src/plugins/map-dxf-export.ts`
- `examples/views/NG/GI/NGGI07.vue`

理由：

当前插件出口是多个子路径，例如：

- `vue-maplibre-kit/plugins/map-feature-snap`
- `vue-maplibre-kit/plugins/line-draft-preview`
- `vue-maplibre-kit/plugins/intersection-preview`
- `vue-maplibre-kit/plugins/map-feature-multi-select`
- `vue-maplibre-kit/plugins/map-dxf-export`

这种设计对 tree-shaking 友好，但新业务开发者需要记忆多个路径。`NGGI07.vue` 为了 snap 示例，需要从 `vue-maplibre-kit/plugins/map-feature-snap` 同时引入 `MapFeatureSnapRule` 和 `createMapFeatureSnapPlugin`。

建议增加 `vue-maplibre-kit/plugins` 聚合入口，保留现有单插件子路径作为高级按需入口。

### 4. 将 `createBusinessPlugins` 提升为业务推荐注册方式

涉及文件：

- `src/MapLibre/facades/businessPreset.ts`
- `src/business.ts`
- `examples/views/NG/GI/NGGI06.vue`
- `examples/views/NG/GI/NGGI07.vue`
- `examples/views/NG/GI/NGGI08.vue`
- `examples/views/NG/GI/NGGI09.vue`
- `examples/views/NG/GI/NGGI10.vue`

理由：

`src/MapLibre/facades/businessPreset.ts` 已提供 `createBusinessPlugins`，并支持 `snap`、`lineDraft`、`intersection`、`multiSelect` 配置。`NGGI06.vue` 已使用该函数完成四插件总览，说明这个方向可行。

但 `NGGI07` 到 `NGGI10` 仍大量使用单插件工厂。建议把 `createBusinessPlugins` 定位为业务层首选方式，单插件工厂保留给需要深度定制插件行为的高级页面。

### 5. 统一业务模拟层组件的类型导入入口

涉及文件：

- `examples/views/NG/GI/components/FeaturePropertyEditor.vue`
- `examples/views/NG/GI/components/NGGI00PopupPanel.shared.ts`

理由：

`FeaturePropertyEditor.vue` 中 `MapFeaturePropertyPanelItem`、`MapFeaturePropertyPanelState` 从 `vue-maplibre-kit` 根入口引入；`NGGI00PopupPanel.shared.ts` 中 `MapSourceFeatureRef` 也从根入口引入。根入口是公开出口，不是内部路径问题。

但从“业务层优先使用 `vue-maplibre-kit/business`”的项目原则看，示例组件若统一从 `vue-maplibre-kit/business` 引入类型，会进一步减少入口分散感。

## 门面优化草案

### 草案一：新增 `vue-maplibre-kit/plugins` 聚合入口

目标是减少业务开发者对多个插件子路径的记忆成本，同时保留现有子路径给高级按需消费场景。

建议使用方式：

```ts
import {
  createBusinessPlugins,
  type MapFeatureSnapRule,
} from "vue-maplibre-kit/plugins";

const plugins = createBusinessPlugins({
  snap: { layerIds: [lineLayerId] },
  lineDraft: true,
  intersection: {
    enabled: true,
    sourceRegistry,
    targetLayerIds: [lineLayerId],
  },
  multiSelect: {
    enabled: true,
    targetLayerIds: [lineLayerId],
  },
});
```

配套调整：

- 新增 `src/plugins.ts` 或 `src/plugins/index.ts` 作为聚合门面。
- `package.json` 增加 `./plugins` export。
- `NGGI07` 到 `NGGI10` 优先展示聚合入口，单插件子路径只在注释或高级示例中出现。

### 草案二：新增业务套件工厂 `createBusinessMapKit`

目标是把业务层最常见的 `mapRef + sourceRegistry + plugins + useBusinessMap` 组合进一步收口。

建议使用方式：

```ts
import { createBusinessMapKit } from "vue-maplibre-kit/business";

const mapKit = createBusinessMapKit({
  mapRef,
  sources: [source],
  plugins: {
    snap: { layerIds: [lineLayerId] },
    lineDraft: true,
    intersection: { enabled: true },
    multiSelect: true,
  },
});

const businessMap = mapKit.map;
const plugins = mapKit.plugins;
```

预期收益：

- 业务页面只需记住一个业务套件入口。
- 示例模板中直接使用 `:plugins="plugins"`。
- 脚本中通过 `businessMap.plugins.lineDraft`、`businessMap.plugins.intersection` 读取状态和动作。

需要注意：

- 不建议隐藏全部 GIS 概念，source、layer、featureRef 仍应保留明确结构。
- 该工厂应作为便利入口，不替代现有低层门面。
