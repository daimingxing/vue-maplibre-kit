# 代码风险与隐患审查报告

审查日期：2026-04-30

审查范围：`src/**` 为主要事实来源，`examples/views/NG/GI/**` 仅用于验证公开 API 的业务接入方式；`docs/**` 不作为代码事实依据。

## 总体判断

当前项目的公开入口、插件宿主、业务门面和示例导入路径整体已经朝 npm 组件库方向收敛，`package.json exports`、Vite 开发别名、TypeScript paths 也都覆盖了根入口、`business`、`config`、`geometry`、`plugins` 和六个插件子路径。

主要风险集中在插件预设校验、failure path 和运行期配置变更：交点插件存在“配置能通过但运行期没有候选线”的路径，部分门面方法会把底层异常直接抛给业务层，部分写回 API 在数据校验失败时仍返回成功，TerraDraw 控件的构造期配置变更也容易让业务层误以为已经生效。

## 高风险：intersection 预设允许缺少候选数据来源

状态：本轮修复。`createBusinessPlugins({ intersection })` 自动模式现在要求 `sourceRegistry`，高级模式可改用 `getCandidates`。

证据：

- `src/MapLibre/facades/businessPreset.ts:272` 到 `src/MapLibre/facades/businessPreset.ts:284`：`resolveIntersectionOptions()` 只校验 `targetSourceIds` 或 `targetLayerIds` 至少存在一个。
- `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:321` 到 `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:327`：自动候选线提取在没有 `sourceRegistry` 时直接返回空数组。
- `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:470` 到 `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:479`：运行期优先使用 `getCandidates`，否则回退到 `sourceRegistry` 自动提取。

判断：

这是已确认的代码风险。业务层可以写出 `createBusinessPlugins({ intersection: { targetLayerIds: [...] } })` 并通过预设校验，但如果既没有顶层或局部 `sourceRegistry`，也没有 `getCandidates`，交点插件运行期不会产生任何候选线。

影响：

页面看起来已经注册 intersection 插件，实际刷新交点时静默无结果。业务层需要回头理解插件内部候选来源规则，排障成本较高。

建议：

预设层校验条件改为 `targetSourceIds || targetLayerIds || getCandidates` 用于判断目标范围，同时再要求 `sourceRegistry || getCandidates` 至少存在其一。错误信息应明确提示“自动模式需要 sourceRegistry，高级模式需要 getCandidates”。

## 高风险：intersection 自动模式声明支持 `MultiLineString`，实际只提取 `LineString`

状态：本轮修复。自动模式会把 `MultiLineString` 拆成 `LineString` 候选线参与交点计算。

证据：

- `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:285` 到 `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:293`：目标图层判断允许 `LineString` 或 `MultiLineString`。
- `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:355` 到 `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:370`：自动提取候选要素时过滤条件只接受 `geometry.type === 'LineString'`。

判断：

这是已确认的不一致。业务层如果把图层声明为支持 `MultiLineString`，该图层会通过目标层判断，但其中的 `MultiLineString` 要素不会进入候选线集合。

影响：

多线要素在业务 source 中“看得见”，但交点计算不参与，容易造成结果缺失。这个问题特别隐蔽，因为类型/图层配置暗示它被支持。

建议：

要么把 `MultiLineString` 展开成多条 `LineString` 候选线并保留来源引用；要么从自动模式的公开说明和目标层判断中移除 `MultiLineString`，只把它留给 `getCandidates` 高级兜底。

## 中风险：`getCandidates` 高级兜底会被业务预设校验挡住

状态：本轮修复。`intersection.getCandidates` 可独立通过业务预设校验，不再要求伪造目标范围。

证据：

- `src/MapLibre/plugins/intersection-preview/types.ts:80` 到 `src/MapLibre/plugins/intersection-preview/types.ts:93`：底层 `IntersectionPreviewOptions` 支持 `getCandidates`。
- `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:470` 到 `src/MapLibre/plugins/intersection-preview/useIntersectionPreviewPlugin.ts:479`：运行时优先调用 `pluginOptions.getCandidates()`。
- `src/MapLibre/facades/businessPreset.ts:272` 到 `src/MapLibre/facades/businessPreset.ts:284`：业务预设仍强制要求 `targetSourceIds` 或 `targetLayerIds`。

判断：

这是已确认的门面校验问题。高级业务方如果只想用 `getCandidates` 提供候选线，仍必须伪造一个 target 配置才能通过 `createBusinessPlugins()`。

影响：

底层插件能力和推荐业务入口不一致，会把高级用户推向单插件工厂，削弱 `createBusinessPlugins()` 的统一入口价值。

建议：

把校验条件放宽为 `targetSourceIds || targetLayerIds || getCandidates`。如果传了 `getCandidates`，可允许不传 target 范围，并在文档中说明此时候选范围完全由业务方负责。

## 高风险：`replaceFeatures()` 校验失败后仍提交快照并返回成功

状态：本轮修复。`replaceFeatures()` 在标准化快照无效时返回 `false`，不提交无效数据。

证据：

- `src/MapLibre/facades/createMapBusinessSource.ts:610` 到 `src/MapLibre/facades/createMapBusinessSource.ts:685`：`normalizeBusinessSourceData()` 会把缺失 ID、重复 ID 等数据标为 `valid: false`。
- `src/MapLibre/facades/createMapBusinessSource.ts:811` 到 `src/MapLibre/facades/createMapBusinessSource.ts:814`：当前快照无效时，`getCurrentFeature()` 直接返回 `null`。
- `src/MapLibre/facades/createMapBusinessSource.ts:939` 到 `src/MapLibre/facades/createMapBusinessSource.ts:945`：`replaceFeatures()` 标准化后不检查 `normalizedSnapshot.valid`，直接 `commitSnapshot(normalizedSnapshot)` 并返回 `true`。

判断：

这是已确认的代码风险。业务层批量替换要素时，只要传入重复业务 ID 或缺失业务 ID，source 会进入无效快照，但调用方得到的仍是成功结果。后续 `resolveFeature()`、属性面板、保存属性等依赖有效索引的能力会降级为不可用。

影响：

- 业务层会误以为批量写回成功。
- UI 可能展示新数据，但后续查询和属性编辑失败，形成“看得见但改不了”的状态。
- 当前返回类型是 `boolean`，缺少失败原因，业务层难以恢复。

建议：

把 `replaceFeatures()` 改为结构化结果，至少在 `normalizedSnapshot.valid === false` 时不提交快照并返回失败原因；如果必须保留 boolean，也应返回 `false` 并暴露 `validationMessage` 给调用方。

## 高风险：运行时 source/layer 动作没有统一包裹 MapLibre 原生异常

状态：本轮修复。运行时 `addSource`、`addLayer`、`removeLayer`、`removeSource` 原生异常会转换成结构化失败结果。

证据：

- `src/MapLibre/facades/useMapLayerActions.ts:200` 到 `src/MapLibre/facades/useMapLayerActions.ts:219`：`addGeoJsonSource()` 直接调用 `rawMap.addSource()`。
- `src/MapLibre/facades/useMapLayerActions.ts:221` 到 `src/MapLibre/facades/useMapLayerActions.ts:255`：`addLayer()` 直接调用 `rawMap.addLayer()`。
- `src/MapLibre/facades/useMapLayerActions.ts:257` 到 `src/MapLibre/facades/useMapLayerActions.ts:281`：`removeLayer()` 和 `removeSource()` 直接调用原生移除方法。

判断：

这是已确认的 failure path 风险。该门面大部分前置检查会返回 `MapLayerActionResult`，但原生 MapLibre 仍可能因样式未加载、source 被 layer 占用、layer spec 非法、source spec 非法等情况抛异常。

影响：

业务层使用的是门面 API，却仍需要记住 MapLibre 原生异常语义；这会破坏“失败都返回结构化结果”的预期。尤其是 `removeSource()`，当 source 仍被图层引用时，MapLibre 通常会抛错，业务页面无法通过返回值处理失败。

建议：

为 `addSource`、`addLayer`、`removeLayer`、`removeSource` 增加 `try/catch`，把异常统一转换成 `MapLayerActionResult(false, message)`。`removeSource()` 可进一步检查仍引用该 source 的 layer，并给出“请先移除图层”的明确提示。

## 中风险：TerraDraw 控件构造期配置变化不会自动重建

状态：已确认设计边界，本轮不改；已补充源码注释、知识库说明和 `docs/problem-record.md` 记录。

证据：

- `src/MapLibre/core/useTerradrawControlLifecycle.ts:145` 到 `src/MapLibre/core/useTerradrawControlLifecycle.ts:154`：挂载 watcher 只跟踪 `isLoaded`、`hasMap`、`isUse`，注释明确说明构造期配置变化不会触发重建。
- `src/MapLibre/core/useTerradrawControlLifecycle.ts:248` 到 `src/MapLibre/core/useTerradrawControlLifecycle.ts:254`：控件只在首次启用时用 `prepareOptions(getConfig())` 创建并 `addControl()`。
- `src/MapLibre/core/mapLibre-init.vue:551` 到 `src/MapLibre/core/mapLibre-init.vue:578`、`src/MapLibre/core/mapLibre-init.vue:586` 到 `src/MapLibre/core/mapLibre-init.vue:613`：`position`、`modeOptions` 和底层控件构造参数都在创建前解析。

判断：

这是已确认的行为边界，也是容易被业务层误解的隐患。`interactive`、`lineDecoration`、`snapping` 有独立 watcher，但 `position`、`modeOptions`、构造参数和部分样式配置不会在 `controls` prop 变化后自动生效。

影响：

业务项目如果提供“页面配置面板”或根据权限动态改变绘图模式，会看到 props 已变化但控件仍沿用旧构造参数。该行为如果不在文档和类型注释里突出说明，排障成本较高。

建议：

明确选择一种契约：要么支持配置变更时通过 `recreateKey` 或显式 API 重建控件；要么在文档中把这些字段标成“仅初始化时生效”，并建议业务层切换 `isUse` 或重新挂载地图容器。

## 中风险：插件初始化失败被宿主吞掉，业务层不容易发现插件缺失

状态：已确认设计边界，本轮不改；保留插件隔离策略，已补充源码注释、知识库说明和 `docs/problem-record.md` 记录。

证据：

- `src/MapLibre/core/useMapPluginHost.ts:695` 到 `src/MapLibre/core/useMapPluginHost.ts:723`：`createPluginRecord()` 捕获插件初始化异常，打印错误后返回 `null`。
- `src/MapLibre/core/useMapPluginHost.ts:812` 到 `src/MapLibre/core/useMapPluginHost.ts:855`：同步插件列表时，创建失败的插件不会进入 `nextPluginRecordMap`。
- `src/MapLibre/facades/mapPluginResolver.ts:48` 到 `src/MapLibre/facades/mapPluginResolver.ts:75`：业务门面后续按插件 type 查找，找不到时返回 `null`。

判断：

这是插件隔离策略带来的隐患，不一定是实现错误。它能避免单插件崩溃拖垮地图，但也会让业务层只看到插件能力不可用，而不是明确知道配置错误或初始化失败。

影响：

外部项目调试时可能表现为 `businessMap.plugins.*` 动作返回失败或无效果，但页面没有可订阅的插件诊断状态。对 npm 组件库来说，这类失败最好能被业务层显式感知。

建议：

保留插件隔离，但给 `MapLibreInitExpose.plugins` 增加 `errors` 或 `diagnostics` 读取能力；也可以触发 `onPluginError` 事件，让业务层在开发环境展示配置错误。

## 中风险：插件配置变更只按顶层引用触发宿主同步

状态：已确认设计边界，本轮不改；已补充源码注释、知识库说明和 `docs/problem-record.md` 记录。

证据：

- `src/MapLibre/core/useMapPluginHost.ts:785` 到 `src/MapLibre/core/useMapPluginHost.ts:796`：宿主只跟踪 `id / type / plugin / options` 顶层依赖。
- `src/MapLibre/core/useMapPluginHost.ts:858` 到 `src/MapLibre/core/useMapPluginHost.ts:864`：插件列表同步由上述依赖触发。
- `examples/views/NG/GI/NGGI12.vue:65` 到 `examples/views/NG/GI/NGGI12.vue:99`：示例通过 `computed(() => createBusinessPlugins(...))` 生成新 descriptor，因此可以正常触发。

判断：

这是可接受的性能取舍，但需要业务层知道“变更插件配置时应替换 options/descriptor 引用”。如果业务代码原地修改嵌套配置对象，宿主未必重新同步依赖。

影响：

同样是修改插件配置，使用 computed 新建描述符能生效，原地修改对象可能不生效，业务层容易把问题归因到插件本身。

建议：

在插件注册文档和类型注释中写清楚：动态配置推荐用 computed 返回新的 `createBusinessPlugins()` 结果，不推荐原地改写嵌套 options。

## 中风险：示例层测试没有进入默认 Vitest 范围

状态：当前代码已包含 `examples/**/*.spec.ts`，该风险已过期。

证据：

- `examples/views/NG/GI/components/NGGI00DemoPanel.spec.ts:1`：examples 下存在组件规格测试。
- `examples/views/NG/GI/components/NGGI00PopupPanel.spec.ts:1`：examples 下也存在 Popup 面板规格测试。
- `vitest.config.ts:79` 到 `vitest.config.ts:82`：默认测试范围只有 `src/**/*.spec.ts`。

判断：

这是已确认的测试覆盖缺口。examples 被项目定义为模拟真实业务引入 npm 包的功能验证页面，但默认 `npm test` 不会执行 examples 下的测试。

影响：

公开入口路径、示例辅助组件和业务体验示例的回归不能被默认测试发现。尤其是 examples 承担“npm 包路径模拟接入”的职责，默认测试不覆盖会降低门面回归保护。

建议：

增加 `test:examples` 脚本，或把关键 examples 规格纳入默认 Vitest include。至少应覆盖 `NGGI00` shared helper、轻量示例的公开入口导入，以及 `createBusinessPlugins + useBusinessMap().plugins.*` 的示例契约。

## 低风险：示例层仍暴露部分插件内部身份字段

状态：已确认公开元数据边界，本轮不改；`generatedKind` 可供业务层识别生成要素类型，已补充知识库说明。

证据：

- `examples/views/NG/GI/NGGI08.vue:76` 到 `examples/views/NG/GI/NGGI08.vue:93`：示例直接读取 `context.generatedKind` 和 `feature.properties?.generatedKind`。
- `examples/views/NG/GI/NGGI00.vue:204` 到 `examples/views/NG/GI/NGGI00.vue:206`、`examples/views/NG/GI/NGGI00.vue:2911` 到 `examples/views/NG/GI/NGGI00.vue:2912`：完整示例使用 `LINE_DRAFT_PREVIEW_SOURCE_ID` 判断线草稿来源。

判断：

这不是直接 bug，但说明业务层在处理临时要素时仍需要知道 `generatedKind`、预览 sourceId 等内部身份。对高级示例可以接受，但如果成为常规业务写法，会增加误用概率。

建议：

在 `businessMap.feature` 或 `businessMap.plugins.lineDraft` 中提供更语义化的判断方法，例如 `isLineDraftFeature()`、`getDraftKind()`、`isDraftFeatureRef()`，减少业务层手写常量判断。
