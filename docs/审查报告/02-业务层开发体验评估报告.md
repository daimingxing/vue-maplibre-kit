# 业务层开发体验评估报告

审查日期：2026-05-11

审查范围：公开入口、业务门面、业务预设、示例页 `examples/views/NG/GI/**`、插件注册与消费路径。

## 总体结论

业务层开发体验整体已经比较清楚：业务页面可以从 `vue-maplibre-kit/business` 获取地图组件、业务 source、图层工厂、样式工厂和 `useBusinessMap()`；插件注册优先从 `vue-maplibre-kit/plugins` 使用 `createBusinessPlugins()`；插件动作统一从 `businessMap.plugins.*` 读取。这个方向符合 npm 组件库“只依赖公开出口完成接入”的原则。

当前体验上的主要问题不是能力缺失，而是“第一屏接入需要理解的对象仍偏多”：业务 source、registry、layers、plugins、mapRef、controls、interactive 都要同时出现。对熟悉 GIS 的业务开发者可接受，对首次接入者仍有一定门槛。

## 体验评分

| 维度 | 评价 | 说明 |
| --- | --- | --- |
| 公开入口清晰度 | 较好 | `business`、`plugins`、`config`、`geometry` 分工明确，导出和别名基本对齐 |
| 插件注册体验 | 较好 | `createBusinessPlugins()` 支持常用插件集中注册，顶层 `sourceRegistry` 可复用 |
| 插件消费体验 | 较好 | `useBusinessMap().plugins.*` 覆盖六个插件，减少业务层记忆多个 `use*` |
| 最小接入体验 | 中等 | 只想使用图层动作或插件状态，也必须准备 `sourceRegistry` |
| 示例可复制性 | 中等偏好 | 小示例比 `NGGI00` 更适合复制，`NGGI00` 信息量过大 |
| 错误反馈 | 较好 | 多数门面返回 `false`、`null` 或结构化结果；少数图层动作仍需补异常边界 |

## 主要发现

### EXP-01：推荐接入路径已经成型

当前推荐路径可以稳定描述为：

```ts
import {
  MapLibreInit,
  MapBusinessSourceLayers,
  createMapBusinessSource,
  createMapBusinessSourceRegistry,
  useBusinessMap,
} from "vue-maplibre-kit/business";
import { createBusinessPlugins } from "vue-maplibre-kit/plugins";
```

源码依据：

- `src/entries/business.ts` 暴露业务组件、source、layer、样式、表达式、`useBusinessMap()` 和兼容 `createBusinessPlugins()`。
- `src/entries/plugins.ts` 暴露 `createBusinessPlugins()` 和六个插件能力。
- `package.json`、`vite.config.ts`、`tsconfig.app.json` 都配置了根入口、`business`、`plugins`、`config`、`geometry` 和六个插件子路径。

这个结构对真实业务项目友好，符合“示例页按 npm 消费方方式导入”的目标。

### EXP-02：`createBusinessPlugins()` 的价值明显，但边界需要继续强调

`createBusinessPlugins()` 不只是少写几个插件工厂名，它还承担了：

- 顶层 `sourceRegistry` 复用，供 `intersection` 和 `dxfExport` 使用。
- `snap.businessLayers` 命名规则简写。
- `lineDraft`、`polygonEdge`、`multiSelect`、`dxfExport` 的 `true` 简写。
- 对 `intersection` 范围缺失、`dxfExport` 缺少 registry 的 fail-fast。

这个体验是正向的。需要继续保持文档里的边界说明：`intersection` 不支持 `true`，因为求交必须有业务范围；`snap: true` 只启用基础吸附能力，业务图层吸附仍需要 `businessLayers`。

### EXP-03：`useBusinessMap()` 聚合门面好用，但 options 对轻量页面偏重

源码中 `UseBusinessMapOptions` 当前要求：

```ts
mapRef: MaybeRefOrGetter<MapLibreInitExpose | null | undefined>;
sourceRegistry: MapBusinessSourceRegistry;
```

这对“标准业务 source + 插件 + 属性编辑”的页面合理。但对只想使用 `businessMap.layers`、`businessMap.effect`、`businessMap.plugins.snap` 或 `businessMap.plugins.multiSelect` 的轻量页面来说，仍必须创建空 registry：

```ts
const sourceRegistry = createMapBusinessSourceRegistry([]);
```

这不是 bug，但会增加首次接入心智负担。

建议后续评估：

- 允许 `sourceRegistry` 可选，内部在未传时创建空 registry。
- 或提供 `createEmptyBusinessSourceRegistry()`，让文档语义更明确。

如果保留必填，也应在“最小地图”和 `useBusinessMap` 总览中明确说明：即使暂时没有业务 source，也可以传空注册表。

### EXP-04：示例体系分层合理，但 NGGI00 不适合作为入门复制模板

当前 `NGGI00.vue` 覆盖地图、控件、source/layer、Popup、属性编辑、绘图测量、snap、lineDraft、intersection、polygonEdge、多选、DXF 和 rawHandles，适合作为完整功能验证页。

更适合业务开发复制的是：

- `NGGI01.vue`：最小地图骨架。
- `NGGI06.vue`：六插件总览，展示 `createBusinessPlugins()` 与 `businessMap.plugins.*`。
- `NGGI07.vue` 到 `NGGI12.vue`：按插件拆分的单能力示例。
- `nggi-example.shared.ts`：业务 source、图层、样式和 mock 数据的可复用组织方式。

建议文档继续把 `NGGI00` 定位为“综合验证页”，不要把它包装成首接入模板。

### EXP-05：业务门面返回值降级策略总体友好

多数插件门面在插件未注册或地图未就绪时返回 `false`、`null`、空集合或结构化失败结果。例如：

- `useMapFeatureSnap()` 未注册时 `resolveMapEvent()` 返回空吸附结果。
- `usePolygonEdgePreview()` 未注册时 `generateFromSelected()` 返回明确失败消息。
- `useMapDxfExport()` 未注册时 `exportDxf()` 返回 `null`。
- `createBusinessPlugins()` 对明显缺失上下文的配置直接抛出中文错误。

这对业务层很友好。建议把 `useMapLayerActions` 的属性设置异常边界补齐后，整体门面一致性会更好。

## 建议优先级

| 优先级 | 建议 | 理由 |
| --- | --- | --- |
| P2 | 让 `useBusinessMap()` 支持无 source 的轻量页面，或提供空 registry 工厂 | 降低首接入门槛 |
| P2 | 补齐 `useMapLayerActions` 属性设置异常边界 | 让业务层拿到稳定结构化结果 |
| P3 | 修正 `NGGI06` DXF 文件名旧语义 | 避免示例叙事和六插件现状不一致 |
| P3 | 在文档中明确 `NGGI00` 是综合验证页，小示例才是推荐复制模板 | 降低新用户误读成本 |

## 结论

当前业务层开发体验已经可以支撑真实项目消费。最值得继续打磨的是“轻量接入第一步”和“文档示例的一致性”，而不是重做现有门面结构。
