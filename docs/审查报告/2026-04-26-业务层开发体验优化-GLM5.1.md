## 二、计划偏差

| 偏差点 | 计划书描述 | 实际实现 | 评价 |
| --- | --- | --- | --- |
| `lineDraft.previewSelectedLine` | `previewSelectedLine({ segmentIndex, extendLengthMeters })` | `previewLine({ lineFeature, segmentIndex, extendLengthMeters, origin })` | 正向偏差。实现更完整，要求传入 `lineFeature` 而非从选中态推断，语义更明确；额外支持 `origin` 回写 |
| `lineDraft.replaceSelectedLineCorridor` | `replaceSelectedLineCorridor({ widthMeters })` | `replacePreviewRegion({ lineFeature, widthMeters })` | 正向偏差。命名从"替换选中线廊"改为"替换预览区域"，更通用且与 `previewLine` 对称 |

**偏差结论：所有偏差均为正向或符合计划前提条件，无需回退。**

**优点：**

1. **聚焦单一主题**：每个示例只关注一个核心场景，代码量控制在 36 ~ 184 行，阅读负担低

## 五、开发简便性评价

### 5.1 改造前 vs 改造后

| 场景 | 改造前 | 改造后 |
| --- | --- | --- |
| 创建控件 | 手动拼装 `MapControlsConfig` 对象 | `createMapControlsPreset("basic")` 一行 |
| 创建插件 | 逐个调用 `createMapFeatureSnapPlugin(...)` 等工厂 | `createBusinessPlugins({ snap: {...}, lineDraft: true, ... })` |
| 声明图层 | 手动构造 `MapBusinessLayerDescriptor[]`，每个子图层写全量配置 | `createLayerGroup({ defaultStyle, layers: [{ type, id, where }] })`，子图层只写差异 |
| 图层显隐 | `rawHandles.map.setLayoutProperty(id, 'visibility', 'none')` | `businessMap.layers.hide("line-layer")` |
| 修改图层样式 | `rawHandles.map.setPaintProperty(id, key, val)` 逐条调用 | `businessMap.layers.setPaint("line-layer", { "line-color": "#f97316" })` |
| 使用多选插件 | 手动 `resolveMapFeatureMultiSelectApi(mapExpose)` + 空值保护 | `businessMap.plugins.multiSelect.activate()` |
| 读取线草稿数据 | 手动 resolve API + 读取 `data.value` | `businessMap.plugins.lineDraft.getData()` |
| 上手学习 | 只能看 NGGI00 综合验证页（~700 行） | NGGI01 最小示例 36 行起步，逐主题递进 |

### 5.2 真实业务项目引入后的开发体验评价

**评价等级：良好（B+）**

**简化效果显著的方面：**

1. **初始化阶段**：`createMapControlsPreset` + `createBusinessPlugins` 让地图 + 控件 + 插件的初始化代码从 30~50 行压缩到 5~10 行
2. **图层声明**：`createLayerGroup` 让多图层声明从每个图层 15~20 行降到 2~3 行差异项
3. **运行时操作**：`businessMap.layers` 和 `businessMap.plugins` 把原来需要 resolve + 空值保护的模式封装为单一调用
4. **学习曲线**：10 个渐进式示例让新人可以 36 行起步，逐步掌握

**仍可改进的方面：**

1. **`useBusinessMap()` 返回结构偏重**：`draft` 和 `plugins.lineDraft` 是同一个对象（引用），`intersection` 和 `plugins.intersection` 同理。两套入口可能造成困惑——新用户不知道该用 `businessMap.draft.clear()` 还是 `businessMap.plugins.lineDraft.clear()`。建议在文档或类型注释中标明推荐路径
2. **`createLayerGroup` 的 `where` 语义**：目前只支持浅层等值匹配（如 `{ kind: "pipe" }`），复杂过滤仍需手写 `filter`。这在计划书中有明确说明，但业务开发者可能在 `where` 无法满足时缺少明确的升级指引
3. **示例与真实项目的差距**：示例使用了 `createExampleKit()` 封装，真实项目需要自行拼装 `mapOptions`、`source`、`registry`、`layers` 等。`NGGI01` 已经是最小可用示例，但 NGGI02+ 的共享封装让"我需要写多少代码"的判断不够直观。建议在示例顶部增加一行注释标注"去掉共享 helper 后的核心业务代码约 X 行"

## 六、问题与风险

### 6.1 已记录问题

- `docs/problem-record.md` 记录了 Windows 环境下 `rg` 不可用的问题，以及 `tsconfig.app.json` 暴露既有类型问题的情况。处理方式合理。

### 6.2 新增风险

| 风险 | 严重度 | 说明 | 建议 |
| --- | --- | --- | --- |
| `draft` 与 `plugins.lineDraft` 双入口 | 低 | 同一对象两种访问路径，新用户可能困惑 | 在 `UseBusinessMapResult` 类型注释中标注推荐使用 `plugins.lineDraft`，`draft` 保留向后兼容 |
| `createBusinessPlugins({ snap: true })` 无吸附目标 | 低 | `true` 简写不生成 `ordinaryLayers` 规则，snap 启用后可能无效果 | 在 JSDoc 中说明 `true` 简写仅启用基础能力，需要吸附图层时应传 `{ layerIds: [...] }` |
| `useMapLayerActions` 中 `any` 类型 | 低 | `getRawMap` 返回 `any`，丢失类型安全 | 后续考虑引入 MapLibre `Map` 类型或定义最小接口 |
| 运行时样式修改与声明式样式冲突 | 中 | `setPaint`/`setLayout` 在图层重建后会被声明式配置覆盖 | 计划书已识别此风险，建议在 NGGI03 中补充注意事项注释 |

## 七、总结

### 完成度

**计划书 7 个实施步骤全部完成，10 个示例全部就位，门面导出、类型导出、单元测试齐全。**

### 开发简便性

本轮改动显著降低了业务层开发者的上手成本：

- **初始化代码量减少 60~70%**（预设工厂）
- **图层声明代码量减少 70~80%**（图层组 + 简单样式）
- **运行时操作从 resolve 模式变为单一调用**（`businessMap.layers` / `businessMap.plugins`）
- **学习入口从 700 行综合页降为 36 行最小示例**

项目打包引入后，业务开发者完成基础功能的推荐路径已经足够简洁。核心改进点（预设工厂、图层组、插件短路径、图层动作）形成了从声明到运行时的完整简化链路。
