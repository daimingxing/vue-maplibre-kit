# 业务层开发体验审查报告

审查范围：以 `src/` 公开封装层和 `examples/` 模拟业务层为主，重点判断外部业务项目是否能用公开出口简单接入。

总体判断：项目已经有清晰的门面方向，`vue-maplibre-kit/business`、`vue-maplibre-kit/plugins`、`useBusinessMap().plugins.*` 这些主路径是对的。但当前首接入示例和部分预设 API 仍偏“能力展示型”，业务开发者第一次照抄时认知负担偏高。

## 主要开发体验问题

### 1. `NGGI00.vue` 作为真实业务模拟页过重，不适合作为首接入样板

影响程度：高

证据：
- `examples/views/NG/GI/NGGI00.vue` 当前 3315 行。
- `examples/views/NG/GI/NGGI00.vue:151-214` 首段 import 同时覆盖地图、弹窗、业务层、几何、插件、Element Plus、mock、资产和多个本地面板。
- `examples/views/NG/GI/NGGI00.vue:1734-1742` 一次性注册 snap、lineDraft、intersection、polygonEdge、multiSelect、dxfExport 六类插件。
- `examples/views/NG/GI/NGGI00.vue:1753-1792` 又把 `businessMap.feature`、`editor`、多个 `plugins.*` 分组拆成本地变量。

体验问题：
这个页面适合作为集成验证场，但不适合作为“业务项目第一份可复制代码”。新业务开发者会同时遇到 source、layer、style、popup、属性编辑、绘图、测量、吸附、多插件、DXF、rawHandles，难以判断最小必需配置是什么。

建议：
保留 `NGGI00.vue` 作为“大而全集成实验室”，另设一个 100 到 200 行左右的推荐首接入示例，只覆盖：
- `MapLibreInit`
- `MapBusinessSourceLayers`
- `createMapBusinessSource`
- `createLayerGroup`
- `createBusinessPlugins`
- `useBusinessMap`

### 2. 控件预设工厂已移除，控件配置回到显式声明

影响程度：高

证据：
- `src/MapLibre/core/mapLibre-init.vue:5-37` 控件实际渲染依赖 `isUse`。
- `examples/views/NG/GI/nggi-example.shared.ts` 示例 helper 已改为直接返回显式 `MapControlsConfig`。
- 文档主路径推荐页面局部 `controls` 或全局 `mapControls`。

体验问题：
旧控件预设名称会让业务层误以为存在一套稳定推荐组合，但真实业务中控件开关、位置、绘图/测量选择通常需要显式判断。保留预设会和全局 `mapControls` 形成两套入口。

建议：
继续使用显式配置。应用级默认值放到 `vue-maplibre-kit/config`：
```ts
setMapGlobalConfig(defineMapGlobalConfig({
  mapControls: {
    MglNavigationControl: { isUse: true, position: 'top-left' },
    MglScaleControl: { isUse: true, position: 'bottom-left' },
    MaplibreMeasureControl: { isUse: true, position: 'top-right' },
  },
}));
```

### 3. `createBusinessPlugins()` 的简写规则不完全一致

影响程度：中

证据：
- `src/MapLibre/facades/businessPreset.ts:107-119` 中 `snap`、`lineDraft`、`polygonEdge`、`multiSelect` 支持布尔或对象；`intersection`、`dxfExport` 只支持对象。
- `src/MapLibre/facades/businessPreset.ts:291-326` 创建逻辑也按这个差异实现。
- `examples/views/NG/GI/NGGI06.vue:66-98` 六插件示例里，部分插件可写 `true`，部分必须写完整对象。

体验问题：
业务开发者需要记每个插件哪些能写 `true`，哪些必须传对象。`intersection` 和 `dxfExport` 确实依赖 `sourceRegistry` 等上下文，但这个依赖如果只在插件局部体现，会让业务层的配置形状不统一。

建议：
考虑给 `createBusinessPlugins()` 增加顶层上下文，例如：
```ts
createBusinessPlugins({
  sourceRegistry,
  snap: { layerIds },
  intersection: { targetLayerIds: layerIds },
  dxfExport: true,
});
```
其中 `intersection` 仍应显式声明 `targetSourceIds` 或 `targetLayerIds`，因为求交范围是页面业务约束，不适合作为无参默认值。`dxfExport: true` 可以复用顶层 `sourceRegistry` 和全局 DXF 默认值。

### 4. 根入口暴露面很宽，容易让业务层不知道该从哪里找能力

影响程度：中

证据：
- `src/entries/root.ts` 大量平铺导出组件、插件类型、交互类型、属性编辑、图层工具、几何交点等能力。
- `src/entries/business.ts:128-160` 已经提供了 `businessSources`、`businessLayers`、`layerStyles`、`mapExpressions` 这类更容易记忆的分组。

体验问题：
根入口能力完整，但太像“全量工具箱”。如果业务开发者从根入口开始，会更容易陷入查找和选择成本。相比之下，业务入口分组更适合真实项目消费。

建议：
文档和示例继续强化默认路径：业务项目先用 `vue-maplibre-kit/business` 和 `vue-maplibre-kit/plugins`，根入口只作为低层或兼容入口。可以在 README 首屏减少根入口平铺 API，优先展示业务入口。

## 可接受但可优化的问题

- `useBusinessMap()` 分组设计清晰，但示例里经常再拆成本地变量。建议在小示例里直接使用 `businessMap.plugins.lineDraft` 等路径，让业务层先建立统一入口记忆。
- 单插件子路径暴露很完整，适合高级场景；普通示例中应尽量少出现常量子路径，避免业务层误以为必须知道内部 source/layer ID。
- `rawHandles` 示例有价值，但应放在“底层逃生通道”示例里，不宜混在首接入页面中。

## 体验较好的设计点

- `package.json` 已提供 `.`、`./business`、`./config`、`./geometry`、`./plugins` 和各插件子路径 exports，外部项目可以按 npm 子路径消费。
- `src/entries/business.ts:128-160` 的分组导出能降低 import 列表记忆负担。
- `src/MapLibre/facades/useBusinessMap.ts:101-115` 把常用插件状态和动作统一收口到 `businessMap.plugins.*`，方向正确。
- `examples/views/NG/GI/nggi-example.shared.ts:307-322` 的 `createExampleKit()` 证明项目已经具备“组合式业务样板”的雏形，适合进一步产品化为推荐接入模板。

## 建议优先级路线

1. 保持控件配置主路径为全局 `mapControls` 与页面显式 `controls`，不要重新引入控件预设工厂。
2. 新增一个极简业务接入示例，把 `NGGI00.vue` 降级为集成验证页。
3. 统一 `createBusinessPlugins()` 的简写体验，至少给依赖 `sourceRegistry` 的插件提供默认 options helper。
4. README 和知识库首屏只推荐业务入口与插件聚合入口，根入口和单插件子路径放到高级用法。
