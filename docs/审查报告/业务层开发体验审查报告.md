# 业务层开发体验审查报告

审查日期：2026-04-26

## 1. 审查结论

项目已经具备比较清晰的 npm 门面基础：业务页可以从 `vue-maplibre-kit/business` 开始接入，插件从 `vue-maplibre-kit/plugins/*` 显式引入，几何工具从 `vue-maplibre-kit/geometry` 引入，全局配置从 `vue-maplibre-kit/config` 引入。

这个方向是合理的，且没有把 `maplibre-gl`、`vue-maplibre-gl`、`terra-draw` 的所有类型都塞进 kit 入口，门面边界没有明显失控。

但从真实业务项目“第一次安装后快速上手”的角度看，当前仍偏向“能力完整但需要记忆较多概念”。最大问题不是缺 API，而是缺一条足够短的推荐接入路径：安装后要怎么引 CSS、怎么注册全局配置、怎么创建 source、怎么挂基础插件、怎么用 `useBusinessMap` 做常见动作，README 还没有给出完整闭环。

## 3. 高优先级体验问题

### 3.1 缺少稳定 CSS 接入说明和出口

业务项目最先遇到的不是 API，而是地图样式。当前 README 的典型导入方式只展示了 TS API，没有告诉业务方如何引入库样式。

证据：

- `README.md:149`
- `README.md:153`
- `README.md:168`

结合打包结果，样式目前只存在于 `dist/assets/vue-maplibre-kit-*.css`，没有稳定公开路径。业务方会卡在“组件能导入，但地图和控件样式不对”的第一步。

建议：

- 提供稳定 `vue-maplibre-kit/style.css` 出口。
- README 第一屏给出完整接入模板。
- 明确是否还需要业务方单独引入底层库 CSS；推荐由 kit CSS 一次性收口。

### 3.2 缺少一个最小可运行的业务模板

当前 README 更像维护者导读，告诉读者去看哪些文件；但真实业务开发者更需要一个可以复制的最小入口。

建议提供 3 个模板：

- `main.ts`：注册全局配置与引入 CSS
- `map-global-config.ts`：全局默认配置
- `BusinessMap.vue`：挂载 `MapLibreInit`、创建一个 source、创建一组图层、启用常用插件

模板应该优先展示推荐路径，不要在首个示例里展示所有高级能力。

### 3.3 当前示例文件过大，不适合作为上手入口

`examples/views/NG/GI/NGGI00.vue` 当前约 2851 行，并包含大量示例日志。它适合做综合验证页，不适合作为新业务项目的首个学习样板。

证据：

- `examples/views/NG/GI/NGGI00.vue`：约 2851 行
- `examples/views/NG/GI/NGGI00.vue:530`
- `examples/views/NG/GI/NGGI00.vue:1220`
- `examples/views/NG/GI/NGGI00.vue:1782`
- `examples/views/NG/GI/NGGI00.vue:3047`

建议：

- 保留 `NGGI00.vue` 作为综合验证页。
- 另外提供 3 到 5 个小型 recipe 示例，例如基础地图、业务 source + 图层、属性编辑、线草稿、DXF 导出。
- 每个 recipe 控制在 150 到 300 行以内。

## 4. 中优先级体验问题

- `examples/views/NG/GI/NGGI00.vue:1150`

- `examples/views/NG/GI/NGGI00.vue:1487`

这些能力本身合理，但上手路径需要再分层。建议给业务层一个“推荐接入顺序”：

1. 先挂地图和全局配置。
2. 再接业务 source 和图层。
3. 再接 `useBusinessMap`。
4. 最后按需加插件。

### 4.2 插件 API 访问方式仍偏底层

插件注册是清晰的，但业务主动调用插件 API 时还需要记住插件 ID，并通过 `mapRef.plugins.getApi(id)` 或特定 resolver 读取。

当前只有 DXF 有较明确的 `resolveMapDxfExportApi`。

建议：

- 为高频插件补齐 resolver，例如 `resolveLineDraftPreviewApi`、`resolveIntersectionPreviewApi`、`resolveMapFeatureSnapApi`。
- 或在 `useBusinessMap()` 里增加 `plugins` 分组，收口常见插件 API。
- 插件 ID 继续保留，但不要让普通业务动作必须记 ID。

### 4.3 Source / Layer 能力强，但需要更高层批量工厂

`createMapBusinessSource()` 的 ID 策略是严谨的，必须在 `promoteId`、`featureIdKey`、`getFeatureId` 中选择一种。

证据：

- `src/MapLibre/facades/createMapBusinessSource.ts:484`
- `src/MapLibre/facades/createMapBusinessSource.ts:694`
- `src/MapLibre/facades/createMapBusinessSource.ts:1076`

这对库的稳定性是好事，但对业务层上手仍有门槛。真实页面往往还要重复写很多 layer 的 `where`、`style`、`propertyPolicy`。

建议后续增加：

- `createLayerGroup()`
- source 默认样式与默认属性治理策略继承
- 按字段值批量生成图层的工具

### 4.4 根入口过宽，业务入口需要更明确的推荐清单

`src/index.ts` 约 346 行，根入口能力很全；`src/business.ts` 约 202 行，已经更适合作为业务入口。

建议：

- README 中明确“业务开发优先只看 `vue-maplibre-kit/business`”。
- 根入口定位为完整能力入口，不作为新手首选。
- 在 `business.ts` 文档中给出“最常用 20 个导出”的清单，避免业务方在完整导出里查找。

## 5. 低优先级体验问题

### 5.1 `BusinessKit` 命名空间用法降低了类型查找直觉

示例中大量使用 `import type * as BusinessKit from "vue-maplibre-kit/business"`，这能减少 import 行数，但新手需要记住 `BusinessKit.xxx` 命名空间。

证据：

- `examples/views/NG/GI/NGGI00.vue:148`
- `examples/views/NG/GI/NGGI00.vue:294`
- `examples/views/NG/GI/NGGI00.vue:316`
- `examples/views/NG/GI/NGGI00.vue:327`

建议：

- 在最小模板里优先使用具名类型导入。
- 综合示例可以继续用命名空间，作为高级写法。

### 5.2 打包产物包含开发演示资产

`npm pack --dry-run` 显示包内包含 `dist/favicon.svg`、`dist/icons.svg`、`dist/demo-map-global-config.d.ts`。

这不是功能错误，但作为 npm 库可以继续收敛发布物，避免业务方误以为这些也是稳定公开能力。

## 6. 建议迭代顺序

1. 先补 CSS 稳定出口和 README 最小接入模板。
2. 再拆小型 recipe 示例，降低第一次阅读成本。
3. 给插件 API 增加高频 resolver 或 `useBusinessMap().plugins` 分组。
4. 增加 layer group / style preset / plugin preset，减少业务页面重复配置。
5. 清理综合示例中的调试日志，把“综合验证页”和“教学示例页”分开。

## 7. 推荐的首个业务接入模板结构

```txt
src/
  main.ts
  map-global-config.ts
  pages/
    BusinessMap.vue
```

业务开发者第一天只需要记住：

- 全局默认值：`vue-maplibre-kit/config`
- 页面主入口：`vue-maplibre-kit/business`
- 插件入口：`vue-maplibre-kit/plugins/*`
- 底层库类型：仍从原包导入，不让 kit 混淆门面

这个边界目前是合理的，后续优化重点应放在“更短的推荐路径”和“更少的重复配置”，而不是把所有底层能力都塞进 kit 根入口。
