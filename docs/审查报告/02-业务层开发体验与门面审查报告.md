# 业务层开发体验与门面审查报告

审查日期：2026-04-30

审查范围：以 `src/entries/**`、`src/MapLibre/facades/**`、`src/MapLibre/core/**`、`package.json`、`vite.config.ts`、`tsconfig.app.json` 为事实来源；用 `examples/views/NG/GI/**` 评估真实业务接入负担。

## 总体判断

项目已经形成较清晰的 npm 库门面：业务页面优先从 `vue-maplibre-kit/business` 接地图、source、layer、`useBusinessMap()`；从 `vue-maplibre-kit/plugins` 注册常用插件；从 `vue-maplibre-kit/config` 配全局默认值；从 `vue-maplibre-kit/geometry` 做纯几何计算。这个方向是合理的。

当前主要体验问题不是“没有门面”，而是“业务层仍要记住一些底层连接点”：同一份 `sourceRegistry` 要传给多个入口，插件配置仍要写 target source/layer，线草稿和生成要素还会泄漏内部标识，`NGGI00` 也更偏高级插件工厂展示，导致初学者的记忆负担偏高。

## 做得较好的部分

### 公开出口与本地开发映射一致

证据：

- `package.json:22` 到 `package.json:66`：导出了根入口、`./business`、`./config`、`./geometry`、`./plugins` 和六个插件子路径。
- `vite.config.ts:46` 到 `vite.config.ts:71`：库构建多入口与 package exports 对齐。
- `vite.config.ts:80` 到 `vite.config.ts:130`：开发态别名支持用包名路径访问本地源码。
- `tsconfig.app.json:7` 到 `tsconfig.app.json:31`：TypeScript paths 覆盖同样的公开路径。

判断：

这符合“真实项目安装后只依赖公开出口完成接入”的组件库目标，也让 examples 能较真实地模拟 npm 包消费。

### `useBusinessMap()` 的聚合方向正确

证据：

- `src/MapLibre/facades/useBusinessMap.ts:123` 到 `src/MapLibre/facades/useBusinessMap.ts:139`：返回值按 `sources / selection / feature / layers / editor / plugins / effect` 分组。
- `src/MapLibre/facades/useBusinessMap.ts:181` 到 `src/MapLibre/facades/useBusinessMap.ts:195`：源码注释明确给出业务层查找入口顺序。
- `src/MapLibre/facades/useBusinessMap.ts:196` 到 `src/MapLibre/facades/useBusinessMap.ts:233`：聚合复用了现有门面，没有让业务层分别 import 多个 `use*`。

判断：

门面设计整体是合理的。它把能力按业务意图组织，而不是把内部目录结构暴露给外部项目。

## 体验问题：`sourceRegistry` 需要在多个地方重复传入

证据：

- `src/MapLibre/facades/useBusinessMap.ts:34` 到 `src/MapLibre/facades/useBusinessMap.ts:40`：`useBusinessMap()` 必须传 `sourceRegistry`。
- `src/MapLibre/facades/businessPreset.ts:127` 到 `src/MapLibre/facades/businessPreset.ts:141`：`createBusinessPlugins()` 也接收顶层 `sourceRegistry`。
- `examples/views/NG/GI/NGGI06.vue:61` 到 `examples/views/NG/GI/NGGI06.vue:68`：业务示例同时把 `kit.registry` 传给 `useBusinessMap()` 和 `createBusinessPlugins()`。

判断：

这是业务层认知负担。用户需要知道 `sourceRegistry` 既是业务门面的数据真相，又是 intersection、dxfExport 等插件的候选数据来源。

影响：

初学者容易漏传其中一处；同一个概念在两个入口重复出现，也削弱了“只记一个门面”的体验。

建议：

提供更高层的组合工厂，例如 `createBusinessMapKit({ mapRef, sources, plugins })` 或 `createBusinessPlugins({ sourceRegistry: businessMap.sources.registry, ... })` 的推荐样板。短期可在文档中明确一句：`sourceRegistry` 是业务 source 的唯一注册表，`useBusinessMap` 用它做读写，插件用它做候选数据读取。

## 体验问题：常用插件配置仍要记 targetSourceIds、targetLayerIds 等底层目标

证据：

- `src/MapLibre/facades/businessPreset.ts:107` 到 `src/MapLibre/facades/businessPreset.ts:114`：intersection 预设仍暴露 `targetSourceIds` 和 `sourceRegistry`。
- `src/MapLibre/facades/businessPreset.ts:272` 到 `src/MapLibre/facades/businessPreset.ts:284`：intersection 缺少 `targetSourceIds` 和 `targetLayerIds` 时直接抛错。
- `examples/views/NG/GI/NGGI06.vue:72` 到 `examples/views/NG/GI/NGGI06.vue:87`：插件总览示例仍需要写 `targetSourceIds`、`targetLayerIds`、`targetLayerIds` 等配置。
- `examples/views/NG/GI/NGGI07.vue:41` 到 `examples/views/NG/GI/NGGI07.vue:76`：snap 示例要理解 `layerIds / geometryTypes / snapTo / priority / tolerancePx` 组合。

判断：

这些字段对 GIS 能力是必要的，但对业务层来说仍偏底层。当前门面已经减少了插件注册数量，却没有完全减少插件目标选择的记忆负担。

建议：

在 `createBusinessPlugins()` 上增加更业务化的简写，例如按业务 source 自动展开常用 line/fill/point 图层，或提供 `targets: kit.targets.lines()` 这类 helper。文档中也应把“最小可用配置”和“精细规则配置”分开，避免初学者第一眼就接触全部字段。

## 体验问题：示例入口存在轻微混用，削弱“业务入口优先”的记忆模型

证据：

- `examples/views/NG/GI/components/FeaturePropertyEditor.vue:108`：业务组件从根入口 `vue-maplibre-kit` 引入 `MapFeaturePropertyPanelItem` 和 `MapFeaturePropertyPanelState` 类型。
- `examples/views/NG/GI/components/NGGI00PopupPanel.shared.ts:7`：业务 helper 从根入口引入 `MapSourceFeatureRef` 类型。
- `src/entries/business.ts:217` 到 `src/entries/business.ts:220`、`src/entries/business.ts:312`：这些类型已经从 `vue-maplibre-kit/business` 暴露。

判断：

根入口是公开出口，所以这不是越界 bug。但 examples 被定义为业务模拟层，最好尽量演示业务推荐路径。否则开发者会困惑：类型到底从 root、business、geometry 还是 plugins 引入。

建议：

把业务模拟层中可从 `business` 入口拿到的业务类型统一改为 `vue-maplibre-kit/business`。根入口保留给通用组件库消费或迁移兼容，不作为业务示例默认写法。

## 体验问题：`NGGI00` 过重，不适合承担第一业务样板

证据：

- 当前 `examples/views/NG/GI/NGGI00.vue` 有 3323 行。
- `examples/views/NG/GI/NGGI06.vue` 有 363 行，覆盖六插件总览。
- `examples/views/NG/GI/NGGI12.vue` 有 311 行，覆盖 polygonEdge 独立流程。

判断：

`NGGI00` 适合作为“大集成验证页”，但不适合作为外部业务开发者的第一份样板。它同时引入 business、geometry、plugins、插件子路径、多个本地组件和 mock 数据，新人很难判断哪些是接入必需、哪些只是 demo 面板。

建议：

文档和 README 应把 `NGGI01`、`NGGI02`、`NGGI06`、`NGGI12` 作为学习路径，把 `NGGI00` 明确定位为集成回归页。必要时新增一个 `NGGI00-lite` 或 README quickstart，只保留地图、source/layer、`useBusinessMap` 和 `createBusinessPlugins` 的最短路径。

## 体验问题：`NGGI00` 主要展示单插件工厂，与推荐业务入口有认知落差

证据：

- `examples/views/NG/GI/NGGI00.vue:208` 到 `examples/views/NG/GI/NGGI00.vue:222`：`NGGI00` 引入多个单插件工厂和插件类型。
- `examples/views/NG/GI/NGGI00.vue:1745` 到 `examples/views/NG/GI/NGGI00.vue:1750`：插件注册逐个调用 `createMapFeatureSnapPlugin()`、`createLineDraftPreviewPlugin()`、`createIntersectionPreviewPlugin()`、`createPolygonEdgePreviewPlugin()`、`createMapDxfExportPlugin()`。
- `examples/views/NG/GI/NGGI06.vue:50` 到 `examples/views/NG/GI/NGGI06.vue:66`：轻量总览示例使用推荐的 `createBusinessPlugins()`。

判断：

`NGGI00` 确实使用 npm 包公开路径，不是内部越界。但它展示的是高级定制写法，和知识库强调的 `createBusinessPlugins + useBusinessMap().plugins.*` 首选路径有落差。

影响：

如果新业务开发者从最大集成页学习，很容易以为常规接入也需要记住每个 `create*Plugin()` 工厂和每个插件 options 类型。

建议：

保留 `NGGI00` 作为高级总装验证页，但在示例索引、README 和知识库首页明确标记：常规业务从 `NGGI06`、`NGGI12` 学，`NGGI00` 用于查复杂联动和高级定制。

## 体验问题：临时要素身份仍泄漏到业务层

证据：

- `examples/views/NG/GI/NGGI08.vue:76` 到 `examples/views/NG/GI/NGGI08.vue:93`：业务示例读取 `generatedKind` 判断草稿类型。
- `examples/views/NG/GI/NGGI00.vue:204` 到 `examples/views/NG/GI/NGGI00.vue:206`：业务页引入 `LINE_DRAFT_PREVIEW_SOURCE_ID`。
- `examples/views/NG/GI/NGGI00.vue:2911` 到 `examples/views/NG/GI/NGGI00.vue:2912`：业务页用 `LINE_DRAFT_PREVIEW_SOURCE_ID` 判断属性面板提示。

判断：

这是高级能力泄漏到业务层的表现。`generatedKind`、预览 sourceId 这些字段有存在价值，但常规业务页最好不需要直接记。

建议：

增加语义 helper：`businessMap.plugins.lineDraft.isDraftFeature()`、`businessMap.plugins.lineDraft.isDraftRef()`、`businessMap.plugins.lineDraft.getFeatureKind()`。这样示例能表达业务意图，而不是教开发者记内部 sourceId。

## 体验问题：插件读取门面按 type 唯一匹配，定制多个同类插件的故事不清晰

证据：

- `src/MapLibre/facades/mapPluginResolver.ts:39` 到 `src/MapLibre/facades/mapPluginResolver.ts:75`：resolver 按插件 type 定位唯一目标，多个同 type 插件会抛错。
- `src/MapLibre/facades/mapPluginResolver.ts:42` 到 `src/MapLibre/facades/mapPluginResolver.ts:46`：`pluginId` 只是历史兼容参数，用于校验命中的唯一插件。

判断：

对常规业务插件预设来说，同类型唯一是合理的。但项目也保留了单插件子路径给高级定制，如果高级用户注册多个同类插件，`useBusinessMap().plugins.*` 的行为边界需要更明确。

建议：

文档中明确：`useBusinessMap().plugins.*` 面向每类插件一个实例；多实例属于高级逃生场景，应直接使用 `mapRef.value?.plugins.getApi(pluginId)` 或新的命名实例 API。

## 体验问题：`createBusinessPlugins` 双入口暴露增加“该从哪引”的记忆负担

证据：

- `src/entries/business.ts:110` 到 `src/entries/business.ts:111`：`business` 入口兼容导出 `createBusinessPlugins`。
- `src/entries/plugins.ts:7` 到 `src/entries/plugins.ts:8`：`plugins` 聚合入口也导出同一函数，并标注为常用业务插件预设工厂。
- `examples/views/NG/GI/NGGI06.vue:50`、`examples/views/NG/GI/NGGI12.vue:42` 到 `examples/views/NG/GI/NGGI12.vue:46`：轻量示例从 `vue-maplibre-kit/plugins` 引入。

判断：

双入口在未发布阶段不是兼容性硬需求。即使保留兼容导出，也应避免示例和文档分裂。

影响：

开发者会反复记忆“插件注册到底从 business 还是 plugins 引入”。这类小分叉积累后，会增加组件库上手时的认知成本。

建议：

文档和示例统一推荐 `vue-maplibre-kit/plugins`。`business` 入口中的兼容导出可以保留，但应标记为兼容或迁移辅助，不作为新业务页面示例。

## 优先改进顺序

1. 统一 examples 的业务类型 import，让业务模拟层尽量只展示 `business / plugins / config / geometry` 四条主路径。
2. 为 `sourceRegistry` 和插件目标配置提供更短的推荐样板。
3. 给临时要素身份增加语义 helper，减少 `generatedKind`、预览 sourceId 在业务层出现。
4. 把 `NGGI00` 定位为集成验证页，把轻量示例作为首接入路径。
5. 明确插件门面对多实例定制的边界。
6. 统一 `createBusinessPlugins` 的文档推荐入口。
