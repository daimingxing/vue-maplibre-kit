# vue-maplibre-kit 开发体验审查报告

## 审查概述

**审查日期**: 2026-04-25  
**审查范围**: 业务示例 NGGI00.vue (3200+ 行)、业务门面 business.ts、配置对象结构  
**审查视角**: 新手业务开发者接入体验

---

## 一、整体评价

### 优点
1. **门面模式执行到位**: 业务层通过 `vue-maplibre-kit/business` 统一入口接入，避免了在根入口平铺查找
2. **类型提示完善**: 所有配置对象都有完整的 TypeScript 类型约束
3. **分组导出清晰**: `businessSources`、`businessLayers`、`layerStyles`、`mapExpressions` 分组合理
4. **高层聚合门面**: `useBusinessMap` 统一收口选择、编辑、草稿、特效等能力

### 核心问题
1. **配置对象冗余度高**: 3200 行示例中大量重复的样板配置
2. **学习曲线陡峭**: 新手需要理解 source/layer/plugin/controls 四层概念
3. **导入路径记忆负担**: 虽然有分组，但仍需记忆多个子路径

## 二、开发体验问题清单（按影响面分级）

### 🔴 高影响问题

#### 问题 1: 插件配置样板过于冗长
**影响**: 每个插件配置需要 50-200 行，业务开发者需要理解大量内部细节

**当前写法** (线草稿插件示例):
```typescript
const lineDraftPreviewPlugin = createLineDraftPreviewPlugin({
  enabled: true,
  styleOverrides: {
    line: {
      layout: {},
      paint: {
        "line-color": createFeatureStateExpression({
          hover: "white",
          default: "#fa8c16",
        }),
        "line-width": createFeatureStateExpression({
          hover: 6,
          default: 5,
        }),
        "line-dasharray": [2, 1.2],
      },
    },
    fill: {
      layout: {},
      paint: {
        "fill-color": "#fa8c16",
        "fill-opacity": 0.18,
        "fill-outline-color": "#ff7a00",
      },
    },
  },
  onHoverEnter: (context) => { /* ... */ },
  onHoverLeave: (context) => { /* ... */ },
  onClick: (context) => { /* ... */ },
  onDoubleClick: (context) => { /* ... */ },
  onContextMenu: (context) => { /* ... */ },
});
```

**理想写法**:
```typescript
// 方案 1: 预设主题
const lineDraftPreviewPlugin = createLineDraftPreviewPlugin({
  theme: 'warning', // 自动应用橙色系样式
  interactions: {
    onClick: (context) => { /* 只写业务逻辑 */ },
  },
});

// 方案 2: 只覆写关键样式
const lineDraftPreviewPlugin = createLineDraftPreviewPlugin({
  style: {
    color: '#fa8c16',
    dashArray: [2, 1.2],
  },
  interactions: { /* ... */ },
});
```

**建议**:
- 提供 3-5 个预设主题 (`primary`, `warning`, `danger`, `success`)
- 简化样式覆写 API，只暴露高频调整项 (color, width, opacity)
- 交互回调改为可选，提供默认行为

#### 问题 2: 图层配置重复度高
**影响**: 每个图层需要重复声明 `propertyPolicy`、`style`、`geometryTypes`

**当前写法**:
```typescript
const primaryBusinessLayers = [
  createCircleBusinessLayer({
    layerId: LAYER_IDS.circle,
    propertyPolicy: {
      fixedKeys: ["mark"],
    },
    style: {
      layout: circleLayout,
      paint: circlePaint,
    },
    geometryTypes: ["Point"],
    where: { mark: "hole" },
  }),
  createCircleBusinessLayer({
    layerId: LAYER_IDS.circleDec,
    propertyPolicy: {
      fixedKeys: ["name", "mark"],
    },
    style: {
      layout: circleLayout,
      paint: circlePaint,
    },
    geometryTypes: ["Point"],
    where: { mark: "dec" },
  }),
  // ... 更多图层
];
```

**理想写法**:
```typescript
// 方案 1: 批量创建同类图层
const primaryBusinessLayers = createLayerGroup({
  source: SOURCE_IDS.primary,
  defaultStyle: circleLayerStyle,
  defaultPolicy: { fixedKeys: ["mark"] },
  layers: [
    { id: 'circle', where: { mark: "hole" } },
    { id: 'circleDec', where: { mark: "dec" }, policy: { fixedKeys: ["name", "mark"] } },
  ],
});

// 方案 2: 约定优于配置
const primaryBusinessLayers = autoCreateLayers({
  source: SOURCE_IDS.primary,
  // 自动按 geometry type 创建图层，按 mark 字段分层
  splitBy: 'mark',
});
```

**建议**:
- 提供 `createLayerGroup` 批量工厂
- 支持图层继承机制，子图层只写差异
- 提供自动分层工具 (按属性字段自动拆分图层)

#### 问题 3: 样式表达式嵌套过深
**影响**: 业务开发者需要理解 `createFeatureStateExpression` + 属性 helper 的组合逻辑

**当前写法**:
```typescript
const { layout: lineLayout, paint: linePaint } = createLineLayerStyle({
  paint: {
    "line-color": createFeatureStateExpression({
      isFlashing: "#ffff00",
      states: {
        demoStyled: "#ec4899",
      },
      selected: "#f97316",
      hover: "#00ff00",
      order: ["isFlashing", "demoStyled", "selected", "hover"],
      default: matchFeatureProperty("id", { line_1: "#ff0000" }, "#0000ff"),
    }),
    "line-width": createFeatureStateExpression({
      states: { demoStyled: 8 },
      selected: 7,
      hover: 6,
      order: ["demoStyled", "selected", "hover"],
      default: 3,
    }),
  },
});
```

**理想写法**:
```typescript
// 方案 1: 预设状态样式
const lineStyle = createLineLayerStyle({
  baseColor: '#0000ff',
  baseWidth: 3,
  states: {
    hover: { color: '#00ff00', width: 6 },
    selected: { color: '#f97316', width: 7 },
    flashing: { color: '#ffff00' },
  },
  propertyRules: {
    color: { line_1: '#ff0000' }, // 简化属性映射
  },
});

// 方案 2: CSS-like 语法
const lineStyle = createLineLayerStyle(`
  line-color: #0000ff;
  line-width: 3px;
  
  &:hover {
    line-color: #00ff00;
    line-width: 6px;
  }
  
  &[id="line_1"] {
    line-color: #ff0000;
  }
`);
```

**建议**:
- 提供状态样式预设对象，避免每个属性单独写表达式
- 简化属性映射语法，支持对象字面量
- 考虑提供 CSS-like DSL (可选，降低 GIS 开发者学习成本)

#### 问题 5: 交互回调分散
**影响**: 同一个业务逻辑 (如打开属性面板) 需要在多处重复

**当前写法**:
```typescript
// TerraDraw 控件配置
MaplibreTerradrawControl: {
  interactive: {
    onClick: (context) => { openTerradrawPopup(context); },
    onContextMenu: (context) => { openTerradrawContextMenu(context); },
  },
},

// Measure 控件配置
MaplibreMeasureControl: {
  interactive: {
    onClick: (context) => { openTerradrawPopup(context); },
    onContextMenu: (context) => { openTerradrawContextMenu(context); },
  },
},

// 普通图层配置
mapInteractive: {
  onClick: (context) => { openMapFeaturePopup(context); },
  onContextMenu: (context) => { openMapFeatureContextMenu(context); },
},
```

**理想写法**:
```typescript
// 方案 1: 统一交互配置
const interactions = createUnifiedInteractions({
  onClick: (context) => {
    if (context.type === 'terradraw') openTerradrawPopup(context);
    else openMapFeaturePopup(context);
  },
  onContextMenu: (context) => {
    if (context.type === 'terradraw') openTerradrawContextMenu(context);
    else openMapFeatureContextMenu(context);
  },
});

// 应用到所有控件和图层
const mapControls = { terradraw: { interactions } };
const mapInteractive = { interactions };

// 方案 2: 默认行为 + 覆写
const mapControls = {
  terradraw: {
    useDefaultInteractions: true, // 自动打开 popup 和属性面板
  },
};
```

**建议**:
- 提供统一交互配置入口，避免三处重复
- 提供默认交互行为 (打开 popup、属性面板)
- 支持全局覆写 + 局部覆写

## 四、错误提示分析

### 优点
- TypeScript 类型约束完善，编译期能发现大部分错误
- 工厂函数返回类型明确，IDE 自动补全友好

### 改进空间
1. **运行时校验不足**: 配置错误只在运行时报错，缺少友好提示
2. **错误信息不够具体**: 如 "插件初始化失败" 没有说明具体原因
3. **缺少配置校验工具**: 没有提供配置对象校验函数

**建议**:
```typescript
// 提供配置校验工具
import { validateMapConfig } from 'vue-maplibre-kit/business';

const validation = validateMapConfig({
  controls: mapControls,
  interactive: mapInteractive,
  plugins: mapPlugins,
});

if (!validation.valid) {
  console.error('配置错误:', validation.errors);
}
```

## 五、常见陷阱

### 陷阱 1: 忘记注册插件到 mapPlugins 数组
**现象**: 插件配置了但不生效  
**原因**: 创建了插件描述符但忘记加到 `mapPlugins` 数组  
**建议**: 工厂函数返回时自动注册，或提供注册检查工具

### 陷阱 2: sourceId 和 layerId 拼写错误
**现象**: 运行时报错 "source not found"  
**原因**: 手动维护字符串 ID 容易拼错  
**建议**: 使用自动生成 ID + 访问器

### 陷阱 3: 样式表达式 order 顺序错误
**现象**: hover 状态被 selected 覆盖  
**原因**: `order` 数组顺序写反了  
**建议**: 提供默认优先级，只在需要时覆写

### 陷阱 4: propertyPolicy 继承不明确
**现象**: 图层级配置没有生效  
**原因**: 不清楚 source 级和 layer 级规则的合并逻辑  
**建议**: 文档明确说明继承规则，提供合并结果查看工具

## 八、总结

### 核心问题
vue-maplibre-kit 的架构设计合理，但**配置复杂度过高**，导致新手学习曲线陡峭。主要体现在：
1. 插件配置需要 50-200 行样板代码
2. 图层配置重复度高，缺少批量工具
3. 样式表达式嵌套过深，难以理解

### 改进方向
1. **提供预设和简化 API**: 80% 场景使用默认配置，20% 场景深度定制
2. **减少手动维护**: 自动生成 ID、自动注册插件、自动分层
3. **统一配置入口**: 避免同一逻辑在多处重复

### 预期效果
实施上述改进后，典型任务代码量可减少 **80%**，新手接入时间可缩短至 **1 小时**。
