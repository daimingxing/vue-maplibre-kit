## 三、发现的潜在问题（非阻塞）

### ⚠️ P2 问题 1: 插件 services 访问未包裹错误边界

**位置**: `useMapPluginHost.ts:516-522, 839-840, 862-863`

**问题描述**:
```typescript
// 单例服务校验
if (pluginRecord.instance.services?.mapSnap) {  // ⚠️ 未捕获访问错误
  serviceProviderMap.mapSnap.push(pluginId);
}
```

**风险**: 如果插件的 `services` getter 抛出异常，会导致宿主崩溃。

**优先级**: P2（中等）- 实际场景中 `services` 通常是静态对象，但理论上存在风险。

**建议修复**:
```typescript
function getPluginService(pluginRecord: MapPluginRecord, serviceName: 'mapSnap' | 'mapSelection') {
  try {
    return pluginRecord.instance.services?.[serviceName] || null;
  } catch (error) {
    console.error(`[MapPluginHost] 插件 '${pluginRecord.descriptorRef.value.id}' 读取 ${serviceName} 服务失败`, error);
    return null;
  }
}
```

### ⚠️ P2 问题 2: 插件 state 访问未包裹错误边界

**位置**: `useMapPluginHost.ts:610-622, 927`

**问题描述**:
```typescript
if (instance.state) {  // ⚠️ 未捕获访问错误
  pluginRecord.stopStateWatch = watch(
    () => instance.state?.value,  // ⚠️ watch 回调中未捕获错误
    (stateSnapshot) => {
      onPluginStateChange?.({ ... });
    }
  );
}
```

**风险**: 
- 如果 `state` getter 抛出异常，会导致插件初始化失败
- watch 回调中访问 `state.value` 抛错会导致响应式系统异常

**优先级**: P2（中等）- `state` 通常是 `ref` 对象，但理论上存在风险。

---

### ⚠️ P2 问题 3: 深冻结可能冻结 Vue 响应式对象

**位置**: `config.ts:164-193`

**问题描述**:
- `deepFreeze()` 会冻结所有对象，包括 Vue 的 `reactive`/`ref` 对象
- 如果业务层在全局配置中传入响应式对象，冻结后会导致响应式失效

**风险**: 当前 `config.ts` 不使用 Vue 响应式，但如果未来有人在配置中传入响应式对象会出问题。

**优先级**: P2（低）- 当前不影响功能，但需要文档说明或增加检测。

**建议修复**:
```typescript
function deepFreeze<T>(obj: T, frozen = new WeakSet()): Readonly<T> {
  // 检测 Vue 响应式对象
  if (obj && typeof obj === 'object' && '__v_isReactive' in obj) {
    console.warn('[deepFreeze] 检测到 Vue 响应式对象，跳过冻结');
    return obj as Readonly<T>;
  }
  // ... 原有逻辑
}
```
