# 项目审查修复计划

**创建日期**：2026-04-25  
**来源报告**：`docs/审查报告/项目全面审查报告.md`、`docs/审查报告/plugin-system-review.md`、`docs/审查报告/封装层门面设计审查报告.md`、`docs/审查报告/开发体验审查报告.md`  
**审查目标**：先核实严重问题是否真实存在，再按批次修复运行时隐患与 npm 包业务接入体验问题。

## 当前判断

1. `插件错误隔离不足` 属实，属于第一批必须修复的问题。
2. `资源泄漏风险` 部分属实；宿主已有统一销毁入口，但部分插件内部 `watch` 没有实例级 `destroy` 清理，不能简单归为“所有插件都完全缺失 destroy”。
3. `business.ts 缺少关键类型导出` 属实，影响业务层从 `vue-maplibre-kit/business` 单入口接入。
4. `useBusinessMap 缺少地图实例空值保护` 的报告表述过重；已验证空地图实例下主要分组不会崩溃，后续只作为回归测试和友好提示优化跟踪。
5. `插件 API 缺少生命周期清理机制` 在“旧 API 引用失效”角度仍需补齐；仅补插件内部 `destroy` 不足以覆盖业务层已经持有 API 引用的场景。
6. `吸附规则 filter 函数没有错误处理` 属实，属于插件交互链路的运行时稳定性问题。
7. `全局配置冻结只冻结顶层` 属实，属于配置快照可被外部嵌套改写的状态一致性问题；暂不展开到配置复杂度和运行时 schema 校验。

## 修复批次

### 第一批：运行时稳定性与业务入口类型

- [x] 核实 `business.ts` 高频业务类型导出缺口。
- [x] 补充 `MapBusinessSource`、`MapBusinessSourceRegistry`、`MapBusinessLayerDescriptor`、`MapFeaturePropertyPanelItem`、`MapSourceFeatureRef`、`MapFeatureId` 到 `vue-maplibre-kit/business`。
- [x] 为插件宿主补充初始化错误隔离：单个插件 `createInstance` 失败时跳过该插件，保留其他插件继续运行。
- [x] 为插件宿主补充运行期方法错误隔离：`getRenderItems`、`getMapInteractivePatch`、`getPluginLayerInteractivePatch`、`resolveSelectedFeatureSnapshot`、`getApi` 失败时返回安全空值。
- [x] 为 `useBusinessMap` 增加空 `mapRef` 回归测试，确认报告中的“未就绪即崩溃”不成立。
- [x] 为 `map-feature-snap` 控制器补充实例级 `destroy()`，停止配置监听并销毁当前吸附绑定。

### 第二批：插件内部清理闭环

- [x] 为 `line-draft-preview` 控制器补充实例级 `destroy()`，停止内部 `watch`，并在插件实例 `destroy` 中调用。
- [x] 为 `map-feature-multi-select` service 补充实例级 `destroy()`，停止内部 `watch`，并在插件实例 `destroy` 中调用。
- [x] 复核 `intersection-preview` 和 `map-dxf-export` 是否需要显式 `destroy()`；当前未发现 watch、地图事件监听、定时器或长期外部资源，不加形式化空方法。
- [x] 补充插件动态移除场景测试，覆盖移除后宿主会调用旧插件实例 `destroy`。

### 第三阶段前闸门：剩余非开发体验 bug 收尾

- [x] 复核 `项目全面审查报告`、`plugin-system-review`、`封装层门面设计审查报告` 中 P0 与运行时稳定性相关条目。
- [x] 为插件宿主返回的插件 API 增加生命周期代理：插件移除后，旧 API 引用断开原始 API 对象，方法调用返回明确错误。
- [x] 为吸附规则 `filter` 增加错误隔离：业务自定义过滤函数抛错时跳过当前候选，不中断鼠标吸附流程。
- [x] 将全局配置快照改为深冻结，避免 `getMapGlobalConfig()` 读出的嵌套配置被外部改写。
- [x] 明确暂缓开发体验 / 配置复杂度问题：插件配置预设、业务接入模板、图层配置重复、门面返回值统一、JSDoc 与文档补全进入后续阶段，不在本轮混入。

### 第三批：业务接入体验

- [ ] 从 `NGGI00.vue` 中提取最小业务接入模板，优先减少首次接入必须理解的配置量。
- [ ] 评估插件配置预设是否值得做成公开 API；只封装业务高频参数，不把底层库完整配置树搬到 kit 门面。
- [ ] 评估业务图层配置重复项，优先通过现有 `businessLayers` / `layerStyles` 工厂收敛，不新增大而全代理。

## 后续原则

- 每批只处理少量高确定性问题，修完后更新本文件状态。
- 报告中的问题必须先复核源码和测试，不把报告措辞直接当事实。
- 业务入口只导出 kit 自己承诺的业务门面类型；底层库类型继续从原包导入，避免混淆门面边界。
