# 交点临时操作层设计文档

## 1. 背景

当前项目已经具备两类较成熟的地图对象处理链路：

- 正式业务源：负责稳定要素、属性治理、写回和门面暴露
- 插件派生源：负责临时结果、独立渲染和交互接入，典型代表是线草稿预览

现阶段业务预期中，地图会出现“对交点进行操作”的需求，但系统里还没有“交点”这种对象。直接把交点做成正式业务实体会过早引入业务语义和写回责任，因此首版更适合把交点定义为临时操作对象。

## 2. 目标与非目标

### 2.1 目标

- 提供一套统一的交点计算能力，支持：
  - 全图业务线两两求交
  - 当前选中线与全图业务线求交
- 将交点渲染为临时点图层，使其具备：
  - 显示
  - 命中
  - 选中
  - 点击
  - 右键动作入口
- 为每个交点生成可复算稳定 ID，便于后续操作链路复用
- 提供交点上下文，支撑后续扩展动作
- 预留“从交点生成正式点要素”的公共出口，但不直接写业务数据

### 2.2 非目标

- 首版不把交点作为正式业务要素纳入业务 source
- 首版不接入属性面板和属性治理
- 首版不处理复杂重合线段场景
- 首版不做自交高级分析
- 首版不做基于视口的增量求交优化

## 3. 核心设计决策

### 3.1 交点对象定位

交点被定义为“临时派生操作对象”，而不是正式业务要素，也不是 TerraDraw 要素。

交点最终以 `Point Feature` 的形式进入地图，但该 `Point Feature` 只承担以下职责：

- 作为地图可操作目标
- 提供交互上下文
- 承接后续动作入口

交点本身不承担正式业务写回责任。

### 3.2 方案选型

本次采用“交点派生插件”方案。

原因如下：

- 最贴合现有架构中“派生对象由插件托管”的思路
- 可以复用当前普通图层交互体系
- 能保持“临时对象”和“正式业务对象”边界清晰
- 后续扩展为正式点要素时，无需推翻当前设计

### 3.3 正式点要素落点策略

当业务需要把交点转成正式点要素时，目标落点固定为：

- 独立业务点 source
- 独立点 layer

禁止以下两种落点方式：

- 写回交点临时图层
- 默认写回原始线 source

原因如下：

- 临时交点图层不应混入正式业务数据
- 线 source 混入点要素会破坏图层语义，增加样式、查询和导出复杂度

## 4. 整体架构

整体实现分为三层。

### 4.1 纯计算层

职责：给定线集合与求交范围，输出标准化交点集合。

输入建议包含：

- 参与求交的业务线集合
- 当前求交范围 `scope`
- 当前选中线引用
- 计算选项
  - 是否包含端点相交
  - 坐标归一化精度
  - 是否忽略自交

输出不是裸坐标数组，而是标准化交点对象数组。每个对象至少包含：

- `intersectionId`
- `point`
- `scope`
- `leftRef`
- `rightRef`
- `leftSegmentIndex`
- `rightSegmentIndex`
- `isEndpointHit`
- `participants`

说明：

- 纯计算层输出的是交点领域对象
- `feature` 字段属于插件层组装后的交互上下文，不在纯计算层中直接生成

该层只负责计算，不负责地图渲染和交互接入。

### 4.2 交点派生插件层

职责：管理交点临时数据源、渲染点图层、接入普通图层交互系统，并向业务层暴露交点 API。

插件内部维护一份临时 `FeatureCollection<Point>`，其角色与线草稿插件中的临时集合一致，但对象类型改为交点。

插件层负责：

- 接收目标业务线 source 与 layer 配置
- 在输入变化时触发求交刷新
- 将计算结果转为地图点图层
- 承接 hover、selected、click、contextmenu 等交互
- 对外返回交点上下文

插件层不负责：

- 写正式业务点 source
- 属性编辑
- 业务表单逻辑

### 4.3 正式点要素落地层

该层不属于交点插件内部逻辑，而是独立公共出口。

建议提供一个公共方法，将交点上下文转换成标准点要素：

- 输入：交点上下文、可选附加属性
- 输出：标准 `Point Feature`

真正写入哪个业务点 source，由业务层决定。

## 5. 交点上下文设计

首版保留以下全部字段。

### 5.1 必备定位字段

#### `intersectionId`

- 含义：交点临时稳定 ID
- 作用：用于唯一定位交点，支持选中、菜单动作和后续命令传参

#### `point`

- 含义：交点坐标
- 作用：用于渲染、定位和生成正式点要素

#### `scope`

- 含义：当前交点的求交来源范围
- 枚举：
  - `all`
  - `selected`
- 作用：供业务层区分交点是如何产生的

### 5.2 参与对象字段

#### `leftRef`

- 含义：参与相交的第一条线的来源引用
- 建议包含：
  - `sourceId`
  - `featureId`
  - `layerId`
- 作用：定位参与交点的原始线

#### `rightRef`

- 含义：参与相交的第二条线的来源引用
- 结构与 `leftRef` 相同
- 作用：定位另一条参与线

### 5.3 线段命中字段

#### `leftSegmentIndex`

- 含义：第一条线中发生求交的线段索引
- 作用：支持后续精细操作，例如高亮命中段、断线、分段

#### `rightSegmentIndex`

- 含义：第二条线中发生求交的线段索引
- 作用：与 `leftSegmentIndex` 对应

### 5.4 类型与展示字段

#### `isEndpointHit`

- 含义：当前交点是否命中了某条线的端点
- 作用：为后续区分“端点连接”和“中段交叉”预留语义

#### `participants`

- 含义：参与相交对象的摘要信息
- 用途：便于业务层直接做提示、菜单和弹层展示

#### `feature`

- 含义：当前交点临时点要素快照
- 用途：为交互链路提供当前点对象本身，减少重复组装

## 6. 临时稳定 ID 规则

交点 ID 被定义为“可复算稳定 ID”，不是业务主键。

其生成规则如下：

1. 先取得两条参与求交的线引用
2. 按稳定规则排序两条线引用，避免 A-B 和 B-A 生成不同 ID
3. 记录两侧命中的 `segmentIndex`
4. 对交点坐标做固定精度归一化
5. 将上述信息拼接为字符串 ID

建议 ID 组成如下：

```text
intersection:${leftSourceId}:${leftFeatureId}:${leftSeg}:${rightSourceId}:${rightFeatureId}:${rightSeg}:${lngKey}:${latKey}
```

说明：

- `coordDigits` 作为可配置项存在
- 当输入线、命中线段和归一化坐标不变时，ID 保持一致
- 当线数据变化导致交点变化时，ID 允许变化

## 7. 刷新时机与性能边界

### 7.1 刷新时机

交点插件仅在输入事实变化时刷新，不因视口变化刷新。

触发刷新条件如下：

- 目标业务线 source 数据变化
- 当前选中线变化
- `scope` 变化
- 交点计算选项变化
- 插件启用状态变化

以下场景不触发刷新：

- 地图缩放
- 地图拖拽
- 底图切换

### 7.2 `selected` 模式规则

在 `selected` 模式下：

- 没有选中线时，返回空交点集
- 当前选中对象不是线时，返回空交点集
- 当前选中的是交点时，不以交点反推求交对象，仍只认真实选中线

### 7.3 性能边界

首版不做“全图所有 source 自动扫描求交”，只处理显式配置的业务线 source。

建议插件配置支持：

- `targetSourceIds`
- `targetLayerIds`
- `canUseFeature`

原因如下：

- 避免点、面、装饰层误参与求交
- 保证交点能力与业务语义对齐
- 控制全图求交成本

### 7.4 首版求交策略

- `all` 模式：候选业务线两两求交，跳过重复配对
- `selected` 模式：当前选中线对候选业务线求交

## 8. 插件配置与公开 API

### 8.1 插件配置

建议插件配置分为四组。

#### 数据范围

- `targetSourceIds`
- `targetLayerIds`
- `scope`

#### 计算规则

- `includeEndpoint`
- `coordDigits`
- `ignoreSelf`

#### 表现配置

- 点样式
- hover 样式
- selected 样式
- 默认是否显示

#### 交互回调

- `onClick`
- `onContextMenu`
- `onChange`

其中 `onChange` 建议返回：

- 当前 `scope`
- 当前交点数量
- 当前是否存在选中交点
- 最近错误状态

### 8.2 插件 API

建议对外暴露以下 API：

- `refresh()`
- `clear()`
- `setScope(scope)`
- `getData()`
- `getById(id)`
- `getSelected()`
- `show()`
- `hide()`

该 API 既支持“只读当前结果”，也支持“把交点作为当前操作目标”。

## 9. 门面接入方式

不建议业务层直接依赖内部插件查询接口。

建议接入方式如下：

1. 插件层负责能力实现
2. facade 层负责稳定访问入口
3. `useBusinessMap()` 对外暴露交点能力

目标是让业务层通过公开门面使用交点功能，而不是直接依赖内部插件细节。

对外使用体验应与现有线草稿能力保持一致，例如：

- `draft` 表示线草稿能力
- `intersection` 表示交点操作能力

## 10. 正式点要素生成链路

生成正式点要素时，调用链建议如下：

1. 用户点击或右键交点
2. 交点插件回传完整交点上下文
3. 业务层决定是否生成正式点要素
4. 调用公共方法将交点上下文转为标准点要素
5. 写入独立业务点 source 与点 layer

该流程明确遵守以下原则：

- 插件不直接写业务数据
- 公共方法只负责对象转换
- 目标业务点 source 由业务层显式决定

## 11. 错误处理

首版建议采用“静默失败 + 可观测状态”策略。

### 11.1 输入不足场景

以下场景返回空结果，不抛异常：

- `selected` 模式下未选中线
- 目标 source 不存在
- 候选数据中没有可用线要素

### 11.2 非法配置场景

以下场景建议仅在开发期给出明确告警：

- `targetSourceIds` 不存在
- `coordDigits` 非法
- `scope` 非法

### 11.3 计算异常场景

以下场景建议保守降级为空集合：

- 某条线几何损坏
- 坐标不合法
- 去重过程异常

### 11.4 可观测状态

建议插件状态中至少暴露：

- `visible`
- `scope`
- `count`
- `selectedId`
- `lastError`

## 12. 测试策略

测试分为三层。

### 12.1 纯计算层测试

重点覆盖：

- 两条普通线相交可得到单点
- 输入顺序变化时交点 ID 保持稳定
- `all` 与 `selected` 的结果范围正确
- 端点相交可正确标记 `isEndpointHit`
- 平行线和不相交线返回空
- 同一点多次命中可正确去重
- 非线要素不会参与计算

### 12.2 插件层测试

重点覆盖：

- source 数据变化可自动刷新
- 当前选中线变化可触发 `selected` 模式刷新
- `clear`、`refresh`、`setScope`、`getById`、`getSelected` 行为正确
- 显隐切换不影响数据正确性
- 点击交点时可回传完整上下文

### 12.3 门面层测试

重点覆盖：

- `useBusinessMap()` 能稳定拿到交点能力
- 业务层不需要直接依赖内部插件 API
- 交点转正式点要素的公共方法可生成合法点要素

## 13. 首版范围裁剪

首版明确包含：

- `all` 与 `selected` 两种求交范围
- 业务线 source 求交
- 临时交点点图层
- hover、selected、click、右键交互
- 完整交点上下文
- 交点转正式点要素的公共方法

首版明确不包含：

- 属性面板接入
- 正式业务点 source 写入封装
- 复杂重合线段处理
- 自交高级分析
- 视口级优化

## 14. 后续扩展方向

在本设计落地后，可以沿同一套交点上下文继续扩展以下能力：

- 高亮参与线
- 从交点生成正式点要素
- 从交点断线
- 从交点拆分线段

这些动作都建立在“交点先成为稳定临时操作对象”的前提上，不需要推翻本次设计。
